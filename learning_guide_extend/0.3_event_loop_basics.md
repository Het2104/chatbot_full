# 0.3 The Event Loop — Deep Technical Learning

---

## 0.3.1 What Is the Event Loop?

The event loop is the core scheduler of Python's `asyncio` system. It is a **single-threaded loop** that:

1. Manages a queue of coroutines (tasks) ready to run
2. Runs them one at a time (cooperatively — not preemptively)
3. When a coroutine hits an `await`, pauses it and runs the next ready task
4. Resumes the paused coroutine when its awaited operation completes

It is called a "loop" because it continuously loops: check for ready tasks → run them → check for I/O completions → run newly ready tasks → repeat.

```
Event Loop Cycle:
┌─────────────────────────────────────────────────┐
│                                                  │
│  1. Pick the next ready coroutine from queue    │
│  2. Run it until it hits `await` or finishes    │
│  3. If `await`: suspend + register I/O callback │
│  4. Check OS for completed I/O (select/epoll)   │
│  5. Mark completed-I/O coroutines as ready      │
│  6. Back to step 1                              │
│                                                  │
└─────────────────────────────────────────────────┘
```

## 0.3.2 How the Event Loop Handles Your WebSocket Request

Let's trace a single WebSocket connection through the event loop:

```
1. New WebSocket connection arrives
   Event Loop → creates coroutine: websocket_chat(websocket, ...)
   
2. websocket_chat starts running
   → publishes to RabbitMQ (quick, returns fast)
   → hits: await asyncio.to_thread(collect_job_response, job_id, timeout)
   
3. Event Loop suspends websocket_chat
   → moves collect_job_response to a thread pool thread
   → event loop is FREE to handle new requests

4. Meanwhile (during those 10 seconds of AI processing):
   Event Loop handles → User B's authentication request
   Event Loop handles → User C's WebSocket connection
   Event Loop handles → Health check pings
   
5. Thread pool thread completes (AI response collected from Redis)
   → Thread signals event loop: "job_id result is ready"
   → Event Loop marks websocket_chat as ready again
   
6. websocket_chat resumes from the await line
   → sends response to client
   → closes WebSocket
```

**Key insight:** The 10-second AI wait happens in a thread, not the event loop. The event loop serves hundreds of other requests during those 10 seconds.

## 0.3.3 The Single-Threaded Nature — Why It Matters

The event loop runs **one coroutine at a time**. This has critical implications:

```python
# BAD: Blocking the event loop — all other users are stuck
@router.get("/bad")
async def bad_route():
    time.sleep(10)  # BLOCKS the entire event loop for 10 seconds!
    return "done"

# GOOD: Yield to event loop
@router.get("/good")
async def good_route():
    await asyncio.sleep(10)  # Suspends the coroutine, others can run
    return "done"
```

**Common blocking operations to avoid in async functions:**
- `time.sleep()` → use `await asyncio.sleep()`
- `requests.get()` → use `httpx.AsyncClient().get()`
- Blocking file I/O → use `asyncio.to_thread(open, ...)`
- Blocking DB queries → wraps in `asyncio.to_thread()` or use async ORM
- `pika` RabbitMQ consume → run in background thread, NOT the event loop

## 0.3.5 asyncio.to_thread() — Under the Hood

When you call:
```python
result = await asyncio.to_thread(blocking_fn, arg1, arg2)
```

Internally this does:
1. Gets the current event loop: `loop = asyncio.get_running_loop()`
2. Gets the default thread pool executor: `loop.run_in_executor(None, ...)`
3. Submits `blocking_fn(arg1, arg2)` to the thread pool
4. Returns a `Future` object
5. The `await` suspends the coroutine until the Future completes
6. When the thread finishes, the Future is marked done, coroutine resumes

The thread pool defaults to `min(32, os.cpu_count() + 4)` threads.

## 0.3.6 Event Loop Access — Getting and Running

```python
import asyncio

# --- Inside an async function (running inside event loop) ---
async def my_fn():
    loop = asyncio.get_running_loop()  # Get the current loop
    task = asyncio.current_task()      # Get the current task

# --- Starting an event loop from synchronous code ---
asyncio.run(my_fn())  # Python 3.7+ — creates, runs, closes loop

# --- uvicorn starts the event loop automatically ---
# uvicorn main:app --host 0.0.0.0 --port 8000
# ↓
# asyncio.run(serve()) internally
```

## 0.3.7 Startup and Shutdown Events

FastAPI's `@app.on_event("startup")` and `@app.on_event("shutdown")` are async hooks that run on the event loop:

```python
# backend/app/main.py
@app.on_event("startup")
async def startup_event():
    # Runs once when event loop starts
    await connect_to_rabbit()  # Would be async if rabbitmq service was async
    chat_worker.start(run_in_thread=True)  # Starts worker in separate thread
    
@app.on_event("shutdown")
async def shutdown_event():
    # Runs once when event loop is stopping
    chat_worker.stop()
    await cleanup_connections()
```

These run before/after the server accepts requests.

## 0.3.8 Common Event Loop Errors and Solutions

| Error | Cause | Solution |
|---|---|---|
| `RuntimeError: This event loop is already running` | Called `asyncio.run()` inside an async context | Use `await` instead of `asyncio.run()` |
| `BlockingError` or frozen server | Called blocking I/O in async function | Wrap in `asyncio.to_thread()` |
| `RuntimeError: no running event loop` | Tried to get loop from sync context | Use `asyncio.run()` to start a loop |
| Coroutine never awaited warning | Created coroutine but didn't `await` it | Add `await` |

## 0.3.9 Interview Questions and Answers

**Q: What is asyncio's event loop and how does it enable concurrency?**

A: The event loop is a single-threaded scheduler that runs coroutines cooperatively. When a coroutine hits an `await` on an I/O operation, it suspends and yields control back to the loop, which picks another ready coroutine to run. Concurrency is achieved by interleaving many coroutines during their I/O wait times, not by running them in parallel.

**Q: Why does calling `time.sleep()` in an async function break everything?**

A: `time.sleep()` is a blocking call — it literally pauses the OS thread. Since the event loop runs on a single thread, pausing that thread means the event loop cannot run ANY other coroutines. The entire server freezes. Solution: use `await asyncio.sleep()` which yields to the event loop instead of blocking.

**Q: How does uvicorn use the event loop?**

A: uvicorn starts an asyncio event loop, registers the ASGI application, and listens for incoming connections via non-blocking I/O (using the OS's `select`/`epoll`/`kqueue`). Each incoming connection becomes a task scheduled on the event loop. Multiple connections are handled concurrently through coroutine interleaving.
