## 1.3.1 REST Principles

REST (REpresentational State Transfer) is an architectural style for distributed hypermedia systems. A RESTful API follows six constraints:

| Constraint | Description | Your API |
|---|---|---|
| Client-Server | Separation of concerns | Frontend (Next.js) ↔ FastAPI |
| Stateless | Each request has all needed info | JWT carries identity, no server session |
| Cacheable | Responses declare cacheability | GET /faqs should have cache headers |
| Uniform Interface | Consistent resource model | `/api/workflows`, `/api/chat/sessions` |
| Layered System | Intermediaries transparent to client | Nginx → FastAPI (client doesn't know) |
| Code on Demand (optional) | Server can send executable code | Not used |

## 1.3.2 HTTP Method Semantics

```
GET    — Read, no side effects, idempotent
         GET /api/workflows/123
         → Returns workflow, call 1000 times = same result

POST   — Create, has side effects, NOT idempotent
         POST /api/chat/message/queue
         → Creates a new chat job, each call creates a new one

PUT    — Full replace, idempotent
         PUT /api/workflows/123
         → Replace entire workflow, calling twice = same state

PATCH  — Partial update, NOT necessarily idempotent
         PATCH /api/workflows/123
         → Update only changed fields

DELETE — Remove, idempotent
         DELETE /api/workflows/123
         → Calling twice: first call succeeds, second returns 404
           But the end state (workflow deleted) is the same

HEAD   — Like GET but no body (check if resource exists)
OPTIONS — Preflight for CORS
```

## 1.3.3 HTTP Status Code Conventions

```
2xx SUCCESS:
    200 OK           — GET/PUT/PATCH success with response body
    201 Created      — POST created a resource (include Location header)
    204 No Content   — DELETE success, nothing to return

4xx CLIENT ERROR:
    400 Bad Request      — Malformed request (Pydantic validation fails)
    401 Unauthorized     — Not authenticated (no/invalid JWT)
    403 Forbidden        — Authenticated but not authorized
    404 Not Found        — Resource doesn't exist
    409 Conflict         — Duplicate resource (unique constraint violated)
    422 Unprocessable    — Valid JSON but semantic errors (FastAPI default)
    429 Too Many Requests — Rate limited

5xx SERVER ERROR:
    500 Internal Server Error — Unexpected exception
    503 Service Unavailable   — Dependency down (RabbitMQ, Redis)
    504 Gateway Timeout       — Upstream timeout
```

**400 vs 422:**
FastAPI uses 422 by default for Pydantic validation errors. Strictly, 400 is more general ("bad request"), 422 means the server understood the request but couldn't process it due to semantic errors.

## 1.3.4 Resource Naming Conventions

```
NOUNS, not verbs:
    ✓  /api/workflows          (collection of workflows)
    ✓  /api/workflows/123      (specific workflow)
    ✓  /api/workflows/123/nodes  (nodes belonging to workflow 123)
    ✗  /api/getWorkflows
    ✗  /api/createWorkflow

Plural for collections:
    ✓  /api/users
    ✓  /api/chat/sessions
    ✗  /api/user
    ✗  /api/session

Hierarchical for relationships:
    /api/workflows/{id}/nodes      — nodes of a specific workflow
    /api/workflows/{id}/nodes/{id} — specific node of a specific workflow

Actions that don't fit CRUD:
    POST /api/chat/message/queue  — "submit to queue" is an action
    POST /api/auth/login          — "login" is an action
    POST /api/auth/logout         — "logout" is an action
    POST /api/workflows/{id}/execute — "execute" is an action
```

## 1.3.5 Versioning

APIs change over time. Breaking changes need versioning to not break existing clients:

```
URL versioning (most visible, common):
    /api/v1/workflows
    /api/v2/workflows  ← v2 adds new fields, v1 still works

Header versioning:
    GET /api/workflows
    API-Version: 2

Query parameter versioning:
    GET /api/workflows?version=2
```

**Your system:** No versioning currently (`/api/` prefix but no version). For production with external clients, URL versioning is recommended.

## 1.3.6 Request and Response Design

**Consistent request bodies (Pydantic schemas):**
```python
# schemas/chat.py
class ChatMessageRequest(BaseModel):
    message: str = Field(..., min_length=1, max_length=4096)
    session_id: str = Field(..., description="Active chat session UUID")
    
class ChatMessageResponse(BaseModel):
    job_id: str
    status: str = "queued"
    estimated_wait_seconds: Optional[int] = None
```

**Consistent error responses:**
```python
class ErrorResponse(BaseModel):
    error: str
    detail: Optional[str] = None
    code: Optional[str] = None    # Machine-readable error code
    
# "QUEUE_UNAVAILABLE", "RATE_LIMITED", "INVALID_SESSION"
```

**Pagination for list endpoints:**
```python
class PaginatedResponse(BaseModel, Generic[T]):
    items: List[T]
    total: int
    page: int
    per_page: int
    has_next: bool

GET /api/workflows?page=2&per_page=20
```

## 1.3.7 Idempotency Keys

For POST operations that should not be duplicated (e.g., payment, job submission):

```
Client adds idempotency key header:
    POST /api/chat/message/queue
    Idempotency-Key: unique-client-uuid-here
    
Server checks if key was seen:
    - Not seen: Process request, store (key → result), return result
    - Already seen: Return previous result without processing again
    
Client retries on timeout:
    POST /api/chat/message/queue
    Idempotency-Key: same-unique-uuid
    → Returns same cached result, no duplicate job
```

## 1.3.8 Interview Questions and Answers

**Q: What is the difference between 401 and 403?**

A: 401 Unauthorized means the client is not authenticated — it hasn't provided credentials or the credentials are invalid/expired. The name is misleading; it really means "unauthenticated." 403 Forbidden means the client IS authenticated but doesn't have permission to access the resource. For example: 401 when a JWT is missing or expired; 403 when a regular user tries to access an admin endpoint.

**Q: Why is POST not idempotent, and why does this matter?**

A: POST creates a new resource each time it's called — calling it twice creates two resources. This matters for retry logic: if a client times out on a POST request, it doesn't know if the server processed it or not. Retrying blindly creates duplicates. Solutions: use idempotency keys, or design endpoints to be naturally idempotent (PUT/PATCH semantics). For your chat queue endpoint, if a user retries a job submission, a duplicate job is created — an idempotency key would prevent this.
