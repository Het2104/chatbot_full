## 4.2.1 Why Services Fail

In a distributed system, failures are not exceptions — they are certainties. Networks partition. Processes crash. Disk fills up. Dependencies slow down. Resilience patterns are design strategies that keep your system functioning (or degrading gracefully) despite failures.

Your system's services can fail independently:
- RabbitMQ broker crashes → queue unavailable
- Redis crashes → pub/sub and cache unavailable
- Milvus (vector DB) unavailable → RAG queries fail
- OpenAI API rate limited → LLM responses fail
- PostgreSQL restarting → user queries fail

Without resilience patterns, one failure cascades into total system failure.

## 4.2.2 Retry with Backoff

The simplest resilience pattern: try again when something fails, but wait progressively longer between attempts.

```python
# chat_worker.py — simplified version of _run_consumer reconnect
import time

def reconnect_with_backoff(max_attempts=10):
    attempt = 0
    delay = 1  # Start with 1 second
    
    while attempt < max_attempts:
        try:
            connection = pika.BlockingConnection(params)
            return connection  # Success
        except pika.exceptions.AMQPConnectionError:
            attempt += 1
            logger.warning(f"Connection attempt {attempt} failed. Retrying in {delay}s")
            time.sleep(delay)
            delay = min(delay * 2, 60)  # Exponential backoff, cap at 60s
    
    raise MaxRetriesExceeded("Could not connect to RabbitMQ")
```

**Your system:** Uses a fixed 5-second retry delay. For production, exponential backoff with jitter prevents thundering herd (all workers retrying simultaneously after broker restart).

**Jitter:**
```python
import random
delay = min(delay * 2, 60) + random.uniform(0, 1)  # Add 0-1s random jitter
```

## 4.2.3 Circuit Breaker

A circuit breaker prevents calling a failing service repeatedly. After a threshold of failures, it "opens" the circuit and returns errors immediately without attempting the call.

```
States:
┌──────────┐  threshold failures  ┌──────────┐  timeout  ┌──────────────┐
│  CLOSED  │ ───────────────────→ │   OPEN   │ ────────→ │ HALF-OPEN   │
│ (normal) │                      │(blocking)│           │  (testing)   │
└──────────┘ ←─────────────────── └──────────┘           └──────────────┘
              success or low                     failure ↑    success ↓
              error rate                                  │    CLOSED
                                                          └───────────
```

**Conceptual implementation:**
```python
class CircuitBreaker:
    def __init__(self, threshold=5, timeout=30):
        self.failures = 0
        self.threshold = threshold
        self.state = "CLOSED"
        self.opened_at = None
        self.timeout = timeout
    
    def call(self, fn, *args):
        if self.state == "OPEN":
            if time.time() - self.opened_at > self.timeout:
                self.state = "HALF-OPEN"
            else:
                raise CircuitOpenError("Service unavailable")  # Fast fail
        
        try:
            result = fn(*args)
            self._on_success()
            return result
        except Exception as e:
            self._on_failure()
            raise
    
    def _on_success(self):
        self.failures = 0
        self.state = "CLOSED"
    
    def _on_failure(self):
        self.failures += 1
        if self.failures >= self.threshold:
            self.state = "OPEN"
            self.opened_at = time.time()
```

**Your system:** Does not currently use circuit breakers. The `is_available()` health check in services acts as a simplistic version — it returns False before attempting operations when the service is known down. A proper circuit breaker would track failure rates and auto-recover.

## 4.2.4 Timeout

Every external call should have a timeout — no call should block indefinitely:

```python
# collect_job_response() — timeout prevents infinite wait
def collect_job_response(self, job_id: str, timeout: int = 30) -> Optional[str]:
    start_time = time.time()
    while time.time() - start_time < timeout:
        # Poll for response...
        time.sleep(0.1)
    return None  # Timed out — return None, caller handles it

# WebSocket handler — uses the timeout return value
result = await asyncio.to_thread(
    pubsub_service.collect_job_response, job_id, WEBSOCKET_RESPONSE_TIMEOUT
)
if result is None:
    await websocket.send_text(json.dumps({"error": "Request timed out"}))
    return
```

**Timeout configuration guidelines:**
- External API calls (OpenAI): 30-60 seconds (LLM is slow)
- Redis operations: 1-5 seconds (should be fast)
- Database queries: 5-30 seconds depending on query complexity
- Internal service calls: 1-10 seconds

## 4.2.5 Bulkhead (Isolation)

Isolate different parts of the system so failure in one doesn't consume resources from another. Named after ship bulkheads that prevent flooding from spreading.

```
Without bulkhead:
    Thread Pool: 20 threads total
    RAG requests hogging all 20 threads → Health check threads starved → 
    Health check times out → Load balancer removes server → All users lose access

With bulkhead:
    Thread Pool A: 15 threads for RAG requests
    Thread Pool B: 5 threads for health checks (reserved, separate)
    Thread Pool C: 3 threads for authentication
    RAG threads full → RAG queues up → Health checks STILL work
```

**Your system uses a natural bulkhead:**
- Event loop thread: Routes/WebSocket (never blocked by worker)
- ChatWorker daemon thread: RabbitMQ consumption (isolated from HTTP)
- Thread pool: Blocking I/O calls (limited by thread pool size)

## 4.2.6 Idempotency — Handling Duplicate Delivery

At-least-once delivery means messages may be processed twice. Idempotent operations produce the same result regardless of how many times they run:

```python
# Idempotent — safe to run twice
def upsert_faq(question, answer):
    existing = db.query(FAQ).filter_by(question=question).first()
    if existing:
        existing.answer = answer  # Update (same result)
    else:
        db.add(FAQ(question=question, answer=answer))  # Create
    db.commit()

# Non-idempotent — dangerous if run twice
def add_points(user_id, points):
    user.total_points += points  # Run twice = double points!
    db.commit()
```

**Your system:** RAG responses are published to Redis channels that are temporary. If the same job is processed twice (duplicate delivery), two responses are published to the same channel. The WebSocket handler reads the first one and the second is ignored (channel expires). This is acceptable for the use case.

## 4.2.8 Interview Questions and Answers

**Q: What is the difference between a retry and a circuit breaker?**

A: Retries attempt the operation again after a failure, useful when failures are transient. A circuit breaker stops attempting after repeated failures, protecting against cascading failures. If you retry indefinitely to a dead service, you flood your thread pool with hanging requests. A circuit breaker detects the service is down and returns errors immediately, freeing threads for other work. They complement each other: retry for occasional failures, circuit breaker for sustained outages.

**Q: Your system uses `is_available()` before each operation — how is this different from a circuit breaker?**

A: `is_available()` checks the current connection state (is the connection object non-None and not closed). It's a static state check, not a failure-rate-based circuit. A proper circuit breaker tracks the rate of failing calls over a time window and opens the circuit proactively before exhausting resources. `is_available()` is simpler but less sophisticated — it only knows about already-detected failures, not about a service that is slow or flaky.
