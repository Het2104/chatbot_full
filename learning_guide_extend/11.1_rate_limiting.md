# 11.1 Rate Limiting — Deep Technical Learning

---

## 11.1.1 Why Rate Limiting Is Essential

Without rate limiting, a single malicious or malfunctioning client can:
- Send 10,000 RAG queries per minute → consume all RabbitMQ queue capacity
- Attempt brute-force password attacks → compromise user accounts
- Enumerate valid user emails via timing attacks → data exposure
- DDoS your system with legitimate-looking traffic → service unavailable for others

Rate limiting caps how many requests a client can make in a time window.

## 11.1.2 Rate Limiting Algorithms

### Token Bucket (Most Common)

```
Bucket capacity: 10 tokens
Refill rate: 1 token/second

State at t=0:  [■■■■■■■■■■] 10 tokens
Request 1:     [■■■■■■■■■□] 9 tokens  → ALLOWED
Request 2:     [■■■■■■■■□□] 8 tokens  → ALLOWED
...10 requests:[□□□□□□□□□□] 0 tokens
Request 11:    [□□□□□□□□□□] 0 tokens  → REJECTED (429)
t=1 second:    [■□□□□□□□□□] 1 token refilled
Request 12:    [□□□□□□□□□□] 0 tokens  → ALLOWED
```

**Pros:** Allows short bursts up to bucket capacity. Natural for users who occasionally exceed the average.
**Cons:** Bucket state must be stored (Redis).

### Fixed Window Counter

```
Window: 1 minute
Limit: 100 requests per window

t=0:00  Counter = 0
t=0:01  Counter = 1  → ALLOWED
...
t=0:30  Counter = 100 → ALLOWED
t=0:31  Counter = 100 → REJECTED
t=1:00  Counter = 0   (window resets)
t=1:00  Counter = 1   → ALLOWED again
```

**Problem:** At t=0:59, all 100 requests come in. At t=1:00, another 100 — 200 requests in 2 seconds.

### Sliding Window Counter

Blends fixed windows to avoid boundary bursts:
```
Current window count + (previous window count × overlap fraction)
```

## 11.1.3 Redis-Based Rate Limiting Implementation

```python
import redis
import time

def check_rate_limit(redis_client, user_id: str, limit: int, window: int) -> bool:
    """
    Token bucket using Redis.
    Returns True if request is allowed, False if rate limited.
    """
    key = f"rate_limit:{user_id}"
    now = time.time()
    
    pipe = redis_client.pipeline()
    
    # Sliding window: count requests in the last `window` seconds
    pipe.zremrangebyscore(key, 0, now - window)    # Remove expired entries
    pipe.zadd(key, {str(now): now})                # Add current request
    pipe.zcard(key)                                # Count requests in window
    pipe.expire(key, window)                       # Set TTL
    
    results = pipe.execute()
    request_count = results[2]
    
    return request_count <= limit
```

**Usage in FastAPI:**
```python
@router.post("/chat/message/queue")
async def queue_message(request: ChatRequest, current_user: User = Depends(get_current_user)):
    
    if not check_rate_limit(redis_client, str(current_user.id), limit=10, window=60):
        raise HTTPException(
            status_code=429,
            detail="Rate limit exceeded. Max 10 messages per minute.",
            headers={"Retry-After": "60"}
        )
    
    # proceed with request
```

## 11.1.4 Rate Limit Granularity

Apply different limits at different levels:

```
Global rate limit (nginx level):
    - 1000 req/s per IP address
    - Protects against DDoS before requests reach Python

Per-endpoint rate limits (application level):
    - /auth/login:         5 attempts/minute per IP (brute force protection)
    - /chat/message/queue: 10 messages/minute per user (queue protection)
    - /health:             unlimited (monitoring tools need this)
    - /admin/*:            20 req/minute per admin user

Per-user rate limits:
    - FREE tier users:     5 RAG queries/day
    - PAID tier users:     unlimited (within global limits)
```

## 11.1.5 Rate Limit Response Headers

Standard headers inform clients how to handle limits:

```
HTTP/1.1 429 Too Many Requests
X-RateLimit-Limit: 10         ← Max requests allowed
X-RateLimit-Remaining: 0      ← Requests left in current window
X-RateLimit-Reset: 1699000060 ← Unix timestamp when limit resets
Retry-After: 45               ← Seconds until client can retry
```

Frontend can use these to:
- Show users how many queries remain
- Auto-retry after the `Retry-After` delay
- Display countdown to reset

## 11.1.6 Nginx Rate Limiting (Pre-Application)

For production, Nginx handles rate limiting before requests reach Python:

```nginx
http {
    # Define rate limit zone: 10 MB state, 10 req/s per IP
    limit_req_zone $binary_remote_addr zone=api_limit:10m rate=10r/s;
    
    server {
        location /api/ {
            limit_req zone=api_limit burst=20 nodelay;
            # burst=20: allow 20 queued, nodelay: don't delay them
            proxy_pass http://fastapi:8000;
        }
        
        location /auth/login {
            limit_req zone=api_limit burst=5;
            proxy_pass http://fastapi:8000;
        }
    }
}
```

## 11.1.7 Rate Limiting for WebSocket

WebSocket connections are long-lived, so the per-request model doesn't directly apply. Instead, rate limit:

1. **Connection rate:** Max N WebSocket connections opened per minute per user
2. **Message rate:** Max N messages sent per connection per time window
3. **Concurrent connections:** Max N simultaneous open connections per user

```python
# Connection-level: check before accept()
@router.websocket("/ws/chat/{session_id}")
async def websocket_chat(websocket: WebSocket, ...):
    
    if not check_concurrent_connections(user_id, max=3):
        await websocket.close(code=1008, reason="Too many connections")
        return
    
    await websocket.accept()
    # ...
```

## 11.1.8 Interview Questions and Answers

**Q: What is the difference between token bucket and sliding window rate limiting?**

A: Token bucket gives clients a burst allowance up to the bucket capacity — a user who hasn't made requests recently can instantly send N requests. Sliding window tracks the exact count of requests in a rolling time window, smoothing out the burst behavior. Token bucket is better for APIs used by humans (occasional bursts are natural behavior). Sliding window is better for machine-to-machine APIs where predictable throughput is important.

**Q: Why use Redis for rate limit state instead of in-memory dictionaries?**

A: In-memory state would be per-process — with multiple FastAPI workers (or multiple servers), each process would have its own counter and the per-user limit would effectively multiply by the number of processes. Redis provides shared, atomic counters accessible from all processes and servers simultaneously. The Redis `INCR` and `ZADD` operations are atomic, preventing race conditions where two requests simultaneously check and increment a counter.
