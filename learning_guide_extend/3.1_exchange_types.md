# 3.1 RabbitMQ Exchange Types — Deep Technical Learning

---

## 3.1.1 What Is an Exchange?

When a producer publishes a message to RabbitMQ, it **never sends directly to a queue**. It sends to an **exchange**. The exchange is a routing engine that decides which queues receive the message.

```
Producer → Exchange → (routing logic) → Queue(s) → Consumer(s)
```

The routing logic depends on the **exchange type**. Your system uses the **direct exchange** (the default, `''`).

---

## 3.1.2 The Four Exchange Types

### 1. Default (Nameless/Direct) Exchange — Used in Your System

```python
# In rabbitmq_service.py
channel.basic_publish(
    exchange='',            # Empty string = default exchange
    routing_key='rag_processing_queue',  # Message goes to queue named exactly this
    body=message_json,
    properties=pika.BasicProperties(delivery_mode=2)
)
```

The default exchange is a built-in direct exchange with no name. When `routing_key` matches a queue name exactly, the message is routed to that queue. Every queue is automatically bound to the default exchange by its name.

```
Producer
    │
    └─→ Default Exchange (routing_key = "rag_processing_queue")
              │
              └─→ Queue: "rag_processing_queue" ✓ (name matches routing_key)
              └─→ Queue: "notifications"         ✗ (name doesn't match)
```

### 2. Direct Exchange

Routes to queues bound with a matching `routing_key`. Multiple queues can have the same binding key.

```
Exchange: "direct_logs"
    │
    ├─→ routing_key="error"   → Queue: error_alerts
    ├─→ routing_key="info"    → Queue: audit_log
    └─→ routing_key="error"   → Queue: dead_letter_processing
                                 (both "error" queues receive the message)
```

Use case: Different severity log levels to different processing queues.

### 3. Topic Exchange

Routes based on **routing key patterns** using `*` (one word) and `#` (zero or more words). Words are separated by dots.

```
Exchange: "topic_exchange"
    │
    ├─→ binding: "rag.*.response"  → Queue: rag_responses
    │                                  matches: "rag.job123.response"
    │                                  no match: "rag.job.batch.response"
    │
    └─→ binding: "chat.#"          → Queue: chat_audit
                                       matches: "chat.user123"
                                       matches: "chat.user123.session.abc"
```

Use case: Fine-grained routing by event type/category/tenant.

### 4. Fanout Exchange

Broadcasts message to **ALL** bound queues, ignoring the routing key entirely.

```
Exchange: "notifications_fanout"
    │
    ├─→ Queue: email_service     (ALL get the message)
    ├─→ Queue: sms_service       (ALL get the message)
    └─→ Queue: push_notification (ALL get the message)
```

Use case: Event broadcasting — one event triggers multiple independent consumers.

## 3.1.3 Headers Exchange

Routes based on **message header attributes** rather than routing key:

```python
channel.basic_publish(
    exchange='headers_exchange',
    routing_key='',  # ignored
    body=message,
    properties=pika.BasicProperties(
        headers={'format': 'pdf', 'type': 'report'}
    )
)
# Queue bound with x-match=all: format=pdf AND type=report
# Queue bound with x-match=any: format=pdf OR type=report
```

Rarely used in practice — topic exchange covers most routing needs.

## 3.1.4 Exchange Durability

Like queues, exchanges can be durable or transient:

```python
# Durable exchange survives broker restart
channel.exchange_declare(
    exchange='my_exchange',
    exchange_type='direct',
    durable=True    # Exchange persists after restart
)

# Transient exchange is lost on restart
channel.exchange_declare(
    exchange='temp_exchange',
    exchange_type='fanout',
    durable=False
)
```

The default exchange is always durable and always exists — no declaration needed.

---

## 3.1.5 Queue Bindings

A binding is the rule connecting an exchange to a queue:

```python
# Declare the queue
channel.queue_declare(queue='my_queue', durable=True)

# Bind it to an exchange with a routing key
channel.queue_bind(
    exchange='direct_logs',
    queue='my_queue',
    routing_key='error'
)
```

**In your system:** The default exchange automatically binds every queue by its own name. You don't need to explicitly call `queue_bind()`.

## 3.1.6 Choosing Exchange Type for Future Features

| Use case | Exchange type | Reason |
|---|---|---|
| Single worker processes a job | Default or Direct | Point-to-point, simple routing |
| Same message to email + SMS | Fanout | Broadcast to all subscribers |
| Route by language/region/tenant | Topic | Pattern matching on dotted keys |
| Route by message content/metadata | Headers | Header attribute matching |

**Current system:** Default exchange (simplest) — one queue, one purpose. If you added a notifications system alongside RAG processing, you'd add a direct or fanout exchange.

## 3.1.7 Dead Letter Exchange (DLX)

When a message is rejected (NACK without requeue), expires, or the queue is full, RabbitMQ can route it to a **dead letter exchange**:

```python
channel.queue_declare(
    queue='rag_processing_queue',
    durable=True,
    arguments={
        'x-dead-letter-exchange': 'dlx_exchange',   # Route failed messages here
        'x-dead-letter-routing-key': 'failed_rag',  # With this routing key
        'x-message-ttl': 3600000,                   # 1 hour TTL
    }
)
```

Failed messages land in a dead letter queue where they can be inspected, retried, or alerted on.

**Your current system:** No DLX configured. Failed messages are NACK'd and discarded, with error response published to Redis so the WebSocket client receives an error message.

## 3.1.8 Interview Questions and Answers

**Q: What is a RabbitMQ exchange and what does it do?**

A: An exchange is a message routing component. Producers publish messages to exchanges, not queues. The exchange examines each message's routing key (and sometimes headers) and forwards the message to one or more bound queues based on its routing rules. This decouples producers from knowing which queues exist.

**Q: When would you use a fanout exchange instead of direct?**

A: When the same event needs to be processed by multiple independent consumers. For example, when a user places an order, you might need to send a confirmation email, update inventory, notify shipping, AND log the event — four independent actions. A fanout exchange broadcasts the "order placed" message to all four bound queues simultaneously without the producer needing to know about each consumer.

**Q: Why does your system use the default exchange with `routing_key = queue_name`?**

A: The system has a single RAG processing queue with a single type of job. Using the default exchange is the simplest approach — you publish with `routing_key='rag_processing_queue'` and the message goes directly to that queue. There's no need for more complex routing logic. If the system needed to route jobs to different specialized workers (e.g., different models for different document types), a direct or topic exchange would be more appropriate.
