## 1.1.1 What Makes a System Distributed?

A distributed system is one where **multiple independent computers cooperate** to appear as a single coherent system to users. Each component runs in its own process (often on separate machines) and they communicate over a network.

Your chatbot system is distributed:
```
FastAPI (app server)  ╮
RabbitMQ (broker)     ╠═ Don't share memory
Redis (cache/pub-sub) ╠═ Communicate via network
Milvus (vector DB)    ╣  Can run on different machines
PostgreSQL (RDBMS)    ╠═ Each has independent failure modes
MinIO (object store)  ╯
```

Compare to a monolith where all components live in the same process and share memory — simpler but less scalable and more fragile.

## 1.1.2 The Eight Fallacies of Distributed Computing

Peter Deutsch (Sun Microsystems) identified assumptions developers incorrectly make:

| Fallacy | Reality | Impact in Your System |
|---|---|---|
| 1. The network is reliable | Networks drop packets, disconnect | RabbitMQ reconnect logic, Redis retry |
| 2. Latency is zero | Every call adds milliseconds | RabbitMQ publish = ~1-5ms overhead |
| 3. Bandwidth is infinite | Networks have limits | Large RAG responses can be chunked |
| 4. The network is secure | Must encrypt, authenticate | JWT on all API calls, TLS in production |
| 5. Topology doesn't change | IPs change, services restart | Service discovery via Docker DNS |
| 6. There is one administrator | Multiple teams | Config management becomes critical |
| 7. Transport cost is zero | Serialization overhead exists | JSON encoding/decoding every message |
| 8. The network is homogeneous | Different protocols/formats | Redis, AMQP, PostgreSQL all different |

## 1.1.3 Fundamental Distributed System Challenges

### Partial Failure
In a single process, it either works or crashes. In distributed systems, **part of the system can fail** while the rest continues:

```
Scenario: Milvus (vector DB) crashes
    ↓
FastAPI is still running → accepts requests
RabbitMQ is still running → queues jobs  
ChatWorker is running → picks up jobs
ChatWorker calls Milvus → ConnectionError
    ↓
Without error handling: Worker crashes, message is NACK'd
With error handling: Worker catches error, publishes "service unavailable" 
                      error response, ACKs message
```

### Ordering
Messages in a distributed system may arrive out of order:
```
Publisher sends: Message1, Message2, Message3
Network reorders: Consumer receives: Message1, Message3, Message2
```

For your RAG pipeline, each query is independent — message ordering doesn't matter. For other use cases (financial transactions, chat history), ordering is critical.

### Consensus
Getting all nodes to agree on something (e.g., who is the leader, what value is canonical) is fundamental and hard. Your system avoids this by design — stateless workers, centralized queue.

## 1.1.4 Synchronous vs Asynchronous Communication Patterns

```
SYNCHRONOUS (request-response):
    Client ──────────→ Server (blocks, waiting)
    Client ←────────── Server (response arrives)
    Client continues
    
    Used in: HTTP REST, gRPC
    Pro: Simple, immediate feedback
    Con: Both parties must be up, client blocked during processing
    
    Example in your system: /auth/login → JWT response (instant)

ASYNCHRONOUS (fire and forget + later notification):
    Client ───────→ Queue (publish, don't wait)
    Client ←── OK (just the enqueue confirmation)
    Client continues doing other things
    
    ... later ...
    
    Worker ←── Queue (delivers message)
    Worker processes
    Worker ──→ Redis (publishes result)
    Client ←── WebSocket (receives result)
    
    Used in: Message queues, event streaming
    Pro: Decoupled, scalable, handles long-running work
    Con: More complex, eventual consistency, harder to debug
    
    Example in your system: RAG query processing (10+ seconds)
```

## 1.1.5 Service Discovery

Services need to find each other. In Docker Compose, service names are DNS names:

```yaml
# docker-compose.yml
services:
  app:
    environment:
      - RABBITMQ_HOST=rabbitmq      # 'rabbitmq' resolves to the broker container
      - REDIS_HOST=redis
      - POSTGRES_HOST=db
  
  rabbitmq:
    image: rabbitmq:3-management
  
  redis:
    image: redis:alpine
```

Docker's internal DNS resolves `rabbitmq` to the container's IP automatically. Services restart and get new IPs — but the DNS name stays the same.

In Kubernetes (production-scale), service discovery is handled by Kubernetes Services + CoreDNS with the same DNS-based pattern.

## 1.1.6 Eventual Consistency vs Strong Consistency

```
STRONG CONSISTENCY:
    After a write, any subsequent read sees the updated value.
    All readers agree on the same value at all times.
    Example: PostgreSQL with ACID transactions
    Cost: Higher latency (must coordinate all replicas before confirming write)

EVENTUAL CONSISTENCY:
    After a write, eventually all readers will see the updated value.
    During the transition, some readers may see old data.
    Example: Redis Cluster replication, Milvus indexing
    Cost: Applications must handle stale reads
```

**In your system:**
- User storage (PostgreSQL): Strong consistency — must have ACID for user data
- FAQ cache (Redis): Eventual consistency is fine — stale FAQ for a few seconds is acceptable
- Vector search (Milvus): May have indexing lag — new documents visible after index is built

## 1.1.7 Interview Questions and Answers

**Q: Why is your chatbot implemented as a distributed system instead of a monolith?**

A: The RAG query processing (embedding search + LLM inference) takes 5-60 seconds. A monolith would either: 1) Block the HTTP thread for 60 seconds (terrible for concurrency), or 2) Use complex async patterns that mix web serving with AI processing in the same codebase. The distributed approach with a message queue lets each component scale independently — add more workers when queries are slow, add more app servers when HTTP traffic increases. The tradeoff is operational complexity (more services to deploy and monitor).

**Q: What is the first fallacy of distributed computing and how does your code address it?**

A: The network is reliable — in practice, networks drop connections, time out, and partition. The code addresses this through: RabbitMQ service reconnect logic (5-second retry loop), Redis service connection checking before every operation, and the ChatWorker's outer reconnect loop that restarts consuming if the RabbitMQ TCP connection drops. Every external service call is wrapped in try/except and failures are handled explicitly.
