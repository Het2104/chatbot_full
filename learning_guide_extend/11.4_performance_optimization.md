## 11.4.1 The Optimization Mindset

**Rule 1: Don't optimize prematurely.** Measure first, optimize second.
**Rule 2: Optimize the bottleneck.** Optimizing a non-bottleneck yields nothing.
**Rule 3: Measure before and after.** "It feels faster" is not a metric.

```
Optimization workflow:
    1. Observe symptoms (p99 latency too high, error rate rising)
    2. Profile to find bottleneck (where is time actually spent?)
    3. Hypothesize root cause
    4. Implement fix
    5. Measure improvement
    6. Repeat from 2 if needed
```

## 11.4.2 Where Time Is Spent in Your System

Profile a typical RAG request to understand where to focus:

```
Total time: ~8,000ms (8 seconds)
├── FastAPI handling: ~5ms         (0.06% — do not optimize)
├── RabbitMQ publish: ~5ms         (0.06% — do not optimize)
├── Queue wait time: ~10ms         (0.1% — depends on worker load)
├── Database query (for context): ~50ms  (0.6%)
├── Milvus vector search: ~200ms   (2.5%)
├── OpenAI LLM inference: ~7,000ms (87.5% — THE BOTTLENECK)
├── Redis publish: ~5ms            (0.06%)
└── WebSocket send: ~2ms           (0.02%)
```

**Conclusion:** Optimizing the Python code yields almost nothing. The bottleneck is the external LLM API call. Options:
- Use a faster model (GPT-3.5-turbo vs GPT-4)
- Stream responses (show tokens as they arrive instead of waiting)
- Cache common questions (FAQ cache is exactly this)

## 11.4.3 Connection Pooling

Every database/Redis connection has a startup cost (TCP handshake, auth). Connection pools reuse established connections:

```python
# WITHOUT pool (bad for performance):
def handle_request():
    conn = psycopg2.connect(...)  # ~5ms: new TCP connection + auth every request
    result = conn.execute(query)
    conn.close()                  # Connection discarded

# WITH pool (default SQLAlchemy behavior):
engine = create_engine("postgresql://...", pool_size=10, max_overflow=20)

def handle_request():
    with engine.connect() as conn:  # ~0.1ms: borrow from pool
        result = conn.execute(query)
    # Connection returned to pool automatically, not closed
```

**Your Redis connection pool:**
```python
# redis_cache_service.py
connection_pool = redis.ConnectionPool(
    host=settings.REDIS_HOST,
    port=settings.REDIS_PORT,
    max_connections=50,           # Up to 50 concurrent Redis connections
    decode_responses=True,
)
self._client = redis.Redis(connection_pool=connection_pool)
```

**Pool sizing:**
- Too small: Requests queue waiting for a free connection (latency spike)
- Too large: Too many DB connections (DB memory exhausted)
- Rule of thumb: `pool_size ≈ worker_count × 2`

## 11.4.4 Caching Strategy

The most impactful optimization for read-heavy workloads:

```
Without cache:     Request → DB query → Response  (50ms)
With cache hit:    Request → Redis GET → Response  (1ms)   50× faster
Cache hit rate 80% → Average latency = 0.8×1ms + 0.2×50ms = 10.8ms
```

**Your FAQ cache (already implemented):**
- Key: `faq:{md5_hash_of_question}`
- TTL: Configurable (e.g., 24 hours for stable FAQ content)
- Invalidation: When FAQ is updated, delete/update the cache key

**Cache warming vs lazy loading:**
```python
# LAZY (current approach): Cache on first miss
def get_faq(question):
    cached = redis.get(key)
    if cached:
        return cached                      # Cache hit: fast
    result = db.query(FAQ).filter(...).first()  # Cache miss: slow
    redis.set(key, result, ttl=86400)
    return result

# EAGER (proactive warming): Pre-load at startup
@app.on_event("startup")
async def warm_cache():
    all_faqs = db.query(FAQ).all()
    for faq in all_faqs:
        redis.set(f"faq:{md5(faq.question)}", faq.answer, ttl=86400)
```

## 11.4.5 Database Query Optimization

```python
# N+1 Query Problem (very common, very slow)
# BAD: 1 query to get workflows + N queries for each workflow's nodes
workflows = db.query(Workflow).all()         # 1 query
for w in workflows:
    nodes = db.query(Node).filter_by(workflow_id=w.id).all()  # N queries!
# 100 workflows = 101 queries

# GOOD: JOIN everything in one query
workflows = (db.query(Workflow)
             .options(joinedload(Workflow.nodes))  # SQL JOIN
             .all())                              # 1 query total

# Index optimization
# If you query workflows by status frequently:
# CREATE INDEX idx_workflows_status ON workflows(status);
# Turns full table scan O(n) into index lookup O(log n)
```

**Slow query detection:**
```python
# SQLAlchemy event listener to log slow queries
from sqlalchemy import event

@event.listens_for(engine, "before_cursor_execute")
def before_cursor_execute(conn, cursor, statement, ...):
    conn._query_start_time = time.time()

@event.listens_for(engine, "after_cursor_execute")  
def after_cursor_execute(conn, cursor, statement, ...):
    duration = time.time() - conn._query_start_time
    if duration > 0.5:  # Log queries taking > 500ms
        logger.warning(f"Slow query ({duration:.3f}s): {statement[:100]}")
```

## 11.4.8 Interview Questions and Answers

**Q: What is the N+1 query problem and how do you fix it?**

A: N+1 occurs when you make 1 query to get a list of objects, then N additional queries to fetch related data for each object. For 100 users, you'd make 101 database roundtrips. Fix using eager loading (JOINs) to fetch all data in 1 query, or batch loading to fetch all related records in 1 additional query. SQLAlchemy's `joinedload()` and `subqueryload()` options implement these patterns automatically.

**Q: How would you optimize the RAG query response time in your system?**

A: Profile first to confirm the LLM call is the bottleneck (it almost certainly is). Then: 1) Implement streaming — send tokens to the WebSocket as they're generated instead of waiting for the complete response, giving perceived sub-second first-token latency. 2) Cache exact or near-exact question matches — an embedding similarity search over previously answered questions can return a cached answer instantly for common questions. 3) Pre-warm the Milvus index — ensure index is loaded in memory before the first query. 4) Use a faster/cheaper model for simple questions and escalate to GPT-4 for complex ones.
