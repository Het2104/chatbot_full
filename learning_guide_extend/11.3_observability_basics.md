## 11.3.1 The Three Pillars of Observability

Observability is the ability to understand the internal state of a system by examining its outputs. It rests on three pillars:

```
┌─────────────────────────────────────────────────────────┐
│  LOGS          │  METRICS         │  TRACES             │
├────────────────┼──────────────────┼─────────────────────┤
│  What happened │  How much / fast │  Why it happened    │
│  Discrete      │  Aggregated      │  Distributed        │
│  events        │  numbers         │  causation          │
├────────────────┼──────────────────┼─────────────────────┤
│  "Job abc123   │  "99th-percentile│  "Request R took    │
│   failed at    │   latency is 8.3s│  3ms auth + 42ms DB │
│   14:23:01"    │   for /ws/chat"  │  + 8s OpenAI"       │
├────────────────┼──────────────────┼─────────────────────┤
│  Tools: Loki,  │  Tools:          │  Tools: Jaeger,     │
│  ELK, Splunk   │  Prometheus,     │  Zipkin,            │
│                │  StatsD, Datadog │  OpenTelemetry      │
└─────────────────────────────────────────────────────────┘
```

You need all three — each answers different questions:
- Logs tell you WHAT happened
- Metrics tell you WHETHER something is happening (at scale)
- Traces tell you WHERE in the chain the slowness/failure is

## 11.3.2 Metrics — Key Concepts

### Metric Types

```
COUNTER: Only increases (reset on restart)
    - Total HTTP requests served: 1_234_567
    - Total RAG jobs processed: 4_521
    - Total authentication failures: 89
    USE WHEN: Counting events, calculating rates (rate(requests_total[5m]))

GAUGE: Can increase or decrease
    - Current queue depth: 47 waiting jobs
    - Currently open WebSocket connections: 23
    - Memory usage: 1.2 GB
    USE WHEN: Measuring current state

HISTOGRAM: Distribution of values
    - Request latency distribution (how many < 100ms, < 1s, < 10s)
    - Response sizes distribution
    USE WHEN: Understanding latency/size distributions (p50, p95, p99)

SUMMARY: Similar to histogram, calculated on client side
    - Less flexible than histogram for aggregation
```

### Your System's Key Metrics to Track

```python
# Example Prometheus metrics using prometheus_client library

from prometheus_client import Counter, Gauge, Histogram

# RAG job metrics
rag_jobs_total = Counter('rag_jobs_total', 'Total RAG jobs', ['status'])
rag_job_duration = Histogram('rag_job_duration_seconds', 'RAG job processing time',
                              buckets=[1, 5, 10, 30, 60])

# Queue metrics
queue_depth = Gauge('rabbitmq_queue_depth', 'Current jobs in queue')
active_consumers = Gauge('rabbitmq_consumers', 'Active workers')

# Redis metrics
cache_hits = Counter('redis_cache_hits_total', 'Cache hits', ['cache_type'])
cache_misses = Counter('redis_cache_misses_total', 'Cache misses', ['cache_type'])

# Usage in worker:
def process_job(job):
    start = time.time()
    try:
        result = run_rag(job)
        rag_jobs_total.labels(status='success').inc()
    except Exception:
        rag_jobs_total.labels(status='failure').inc()
    finally:
        rag_job_duration.observe(time.time() - start)
```

## 11.3.3 Distributed Tracing

A single user request touches multiple services. Tracing tracks the full path:

```
Request ID: trace-abc-123

Span 1: FastAPI /ws/chat    [0ms ─────────────────────── 8400ms]
  Span 2: JWT validation    [0ms ── 3ms]
  Span 3: RabbitMQ publish  [3ms ──── 8ms]
  Span 4: Redis subscribe   [8ms ─────────────────────── 8395ms]
        Span 5: DB query    [wait] [600ms ──── 800ms]
        Span 6: Milvus      [wait] [800ms ──────────── 2800ms]
        Span 7: OpenAI API  [wait] [2800ms ───────────── 7800ms]
        Span 8: Redis pub   [wait] [7800ms ─── 7810ms]
  Span 9: WS send result   [8395ms ─ 8400ms]
```

**Trace ID:** A unique identifier passed through ALL services (HTTP headers, message properties) to correlate related spans.

**Span:** A single operation with a start time, duration, and metadata.

## 11.3.4 OpenTelemetry — The Standard

OpenTelemetry (OTel) is the vendor-neutral standard for instrumentation:

```python
# Installation
pip install opentelemetry-sdk opentelemetry-exporter-otlp
pip install opentelemetry-instrumentation-fastapi
pip install opentelemetry-instrumentation-sqlalchemy

# Auto-instrumentation of FastAPI
from opentelemetry.instrumentation.fastapi import FastAPIInstrumentor
from opentelemetry.sdk.trace import TracerProvider

tracer_provider = TracerProvider()
FastAPIInstrumentor.instrument_app(app, tracer_provider=tracer_provider)
# Automatically creates spans for every request
```

**Propagation through RabbitMQ:**
```python
# Publisher: inject trace context into message headers
from opentelemetry import propagate
headers = {}
propagate.inject(headers)  # Adds traceparent header
channel.basic_publish(..., properties=pika.BasicProperties(headers=headers))

# Consumer: extract trace context from message headers
context = propagate.extract(message_headers)
with tracer.start_as_current_span("process_rag_job", context=context):
    # This span is linked to the original HTTP request trace
    result = run_rag_pipeline(query)
```

## 11.3.5 Golden Signals (Google SRE)

Google SRE promotes monitoring four fundamental signals:

| Signal | Description | Your System Metric |
|---|---|---|
| **Latency** | Time to service a request | RAG job duration p50/p95/p99 |
| **Traffic** | Request volume | RAG jobs/minute, WebSocket connections/minute |
| **Errors** | Rate of failed requests | Failed jobs / total jobs (%) |
| **Saturation** | How full the system is | Queue depth, memory usage, active threads |

An alert fires when:
- p99 latency > 60s (job taking too long)
- Error rate > 5% (5% of jobs failing)
- Queue depth > 100 (workers can't keep up)
- Memory > 85% (approaching OOM)

## 11.3.6 Alerting Strategy

Bad alerting: Too many low-priority alerts → oncall fatigue → real alerts ignored.

```
SEVERITY LEVELS:
Page (immediate action):
    - Error rate > 10% for 5 minutes
    - Health endpoint returning 500 for 2 minutes
    - Queue depth > 500 (massive backlog)

Slack notification (business hours):
    - p99 latency > 30s
    - Cache hit rate < 50%
    - Worker reconnect count > 5 in 1 hour

Weekly report (no immediate action):
    - Average job duration trend
    - Cache hit rate trend
    - Daily active users
```

## 11.3.7 Interview Questions and Answers

**Q: What is the difference between monitoring and observability?**

A: Monitoring is checking known metrics against predefined thresholds — you know what to look for (e.g., CPU > 80%, error rate > 5%). Observability is the ability to understand UNKNOWN failures by examining outputs — you can ask arbitrary questions about system state. A highly observable system has rich structured logs, metrics, and traces that let you diagnose issues never anticipated during design. Monitoring tells you something is broken; observability helps you understand WHY.

**Q: Why do you need distributed tracing in addition to logs?**

A: Logs from individual services are siloed — the FastAPI log shows "request received" and "response sent," the worker log shows "job processed." Without a shared trace ID linking these, you can't correlate which API request corresponds to which worker execution when there are thousands of concurrent operations. Distributed tracing links spans across service boundaries, showing the complete causal chain of a single request across all services with exact timing at each step.
