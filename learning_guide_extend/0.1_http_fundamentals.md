# 0.1 HTTP Fundamentals — Deep Technical Learning

---

## 0.1.1 Why This Comes First

Every request to your FastAPI backend travels over HTTP. Before understanding WebSocket upgrades, JWT headers, REST endpoints, or CORS, you must understand HTTP itself — the foundation everything else is built on.

---

## 0.1.2 What is HTTP?

**HTTP (HyperText Transfer Protocol)** is a stateless, request-response application protocol that runs over TCP. A client sends a request message; the server sends a response message; the connection closes (or is reused via keep-alive).

**Stateless** means: each request is independent. The server has no memory of previous requests unless you explicitly pass state (e.g., a JWT token in the Authorization header).

---

## 0.1.3 HTTP Request Structure

```
POST /chat/message/queue HTTP/1.1        ← Request Line: method, path, version
Host: localhost:8000                      ← Headers start
Authorization: Bearer eyJhbGci...
Content-Type: application/json
Content-Length: 52
                                          ← Blank line separates headers from body
{"chatbot_id": 7, "message": "Hello"}    ← Body (payload)
```

**Key components:**
- **Method:** GET (read), POST (create), PUT (replace), PATCH (update), DELETE (remove)
- **Path:** The resource identifier (`/chat/message/queue`)
- **Headers:** Metadata about the request (auth, content type, encoding)
- **Body:** Data payload (only for POST, PUT, PATCH)

---

## 0.1.4 HTTP Response Structure

```
HTTP/1.1 202 Accepted                    ← Status Line: version, code, reason
Content-Type: application/json
Content-Length: 78
                                          ← Blank line
{"session_id": 42, "job_id": "550e84...", "status": "queued"}
```

**Status code categories:**

| Range | Category | Examples |
|---|---|---|
| 2xx | Success | 200 OK, 201 Created, 202 Accepted |
| 3xx | Redirect | 301 Moved Permanently, 304 Not Modified |
| 4xx | Client Error | 400 Bad Request, 401 Unauthorized, 404 Not Found, 422 Unprocessable Entity |
| 5xx | Server Error | 500 Internal Server Error, 503 Service Unavailable |

**In your system:**
- `200` — successful GET (fetching chatbot, FAQs, sessions)
- `201` — resource created (new chatbot, new FAQ)
- `202` — job accepted and queued (POST /chat/message/queue — processing deferred)
- `401` — invalid or missing JWT token
- `422` — Pydantic validation failed (wrong request body shape)

---

## 0.1.5 HTTP Methods and Their Semantics

| Method | Idempotent | Safe | Use in Your System |
|---|---|---|---|
| GET | Yes | Yes | Fetch chatbots, workflows, messages |
| POST | No | No | Login, send message, create chatbot |
| PUT | Yes | No | Replace a workflow entirely |
| PATCH | No | No | Update specific fields of a node |
| DELETE | Yes | No | Delete a chatbot, workflow, FAQ |

**Idempotent:** calling the same request multiple times produces the same result (safe to retry).  
**Safe:** calling does not modify server state.

## 0.1.6 CORS — Cross-Origin Resource Sharing

Your Next.js frontend runs on `http://localhost:3000`. Your FastAPI runs on `http://localhost:8000`. These are **different origins** (different port). Browsers block cross-origin requests by default — this is the **Same-Origin Policy**.

**CORS solves this by adding response headers:**

```
Access-Control-Allow-Origin: http://localhost:3000
Access-Control-Allow-Methods: GET, POST, PUT, DELETE, OPTIONS
Access-Control-Allow-Headers: Authorization, Content-Type
```

When a browser sends a non-simple request (e.g., POST with Authorization header), it first sends a **preflight** `OPTIONS` request to ask "can I do this?" The server must respond with appropriate CORS headers, or the browser blocks the actual request.

**In your `main.py`:**
```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=CORS_ALLOWED_ORIGINS,  # e.g., ["http://localhost:3000"]
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

## 0.1.7 HTTP Keep-Alive and Connection Reuse

HTTP/1.1 introduced **persistent connections** — the TCP connection stays open after one response so the next request can reuse it without the overhead of a new TCP handshake.

**TCP handshake cost:** ~1 round-trip time (RTT). At 100ms network latency, that's 100ms overhead per new connection. With keep-alive, 100 requests share one connection.

FastAPI (via uvicorn) handles this automatically.

---

## 0.1.8 HTTP vs WebSocket — The Upgrade

HTTP is request-response; once the server responds, communication is done. For real-time streaming, you need a persistent bidirectional channel — that's WebSocket.

The WebSocket handshake starts as an HTTP request:
```
GET /ws/chat/42 HTTP/1.1
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
```

The server responds with 101 and the TCP connection is "upgraded" — it stays open using WebSocket framing instead of HTTP. This is why WebSocket requires an HTTP server at first.

## 0.1.9 HTTP in Your Request Flow

```
1. POST /auth/login           HTTP/1.1  → Returns JWT
2. POST /chat/message/queue   HTTP/1.1  → Returns job_id (202 Accepted)
3. GET  /ws/chat/42/job-uuid  HTTP/1.1  → Upgrade to WebSocket (101)
4. WebSocket frames ←→ FastAPI          → AI response delivered
```

Steps 1 and 2 are pure HTTP (request-response). Step 3 starts as HTTP and upgrades. Understanding HTTP is required to understand all three.

## 0.1.10 Interview Questions and Answers

**Q: What does HTTP stateless mean and how does your system maintain user state?**

A: HTTP stateless means each request has no memory of prior requests. Your system maintains user state via JWT tokens — the client sends the token on every authenticated request, and the server extracts user identity from the token without any server-side session storage.

**Q: What is the difference between 401 and 403?**

A: 401 Unauthorized means the client is not authenticated — no valid token provided or token expired. 403 Forbidden means the client IS authenticated but does not have permission to access the resource. Your system returns 401 for missing/invalid JWT.

**Q: What is a CORS preflight request?**

A: A browser-initiated `OPTIONS` request sent before a non-simple cross-origin request (e.g., POST with custom headers). The browser asks the server if the actual request is allowed. The server responds with `Access-Control-Allow-*` headers. If the headers permit the request, the browser proceeds; otherwise it blocks the request without ever sending it.
