## 3.2.1 A Message's Journey — Complete State Machine

Every message published to RabbitMQ passes through a defined state machine. Understanding this is critical for building reliable systems.

```
┌────────────────────────────────────────────────────────────┐
│                    MESSAGE LIFECYCLE                       │
│                                                            │
│  1. Published  → 2. In Exchange → 3. Queued               │
│                                                            │
│  Producer                Exchange            Queue         │
│  [basic_publish] ──────→ [routes] ─────────→ [buffered]  │
│                                                            │
│                                   ↓                        │
│                       4. Delivered to Consumer             │
│                          [Unacknowledged]                  │
│                                                            │
│                           ↓              ↓                 │
│                      5a. ACK         5b. NACK              │
│                      [Deleted]    [Requeue or DLX]         │
│                                                            │
│  Alternative paths:                                        │
│  3 → TTL expires → DLX or discarded                       │
│  3 → Queue full → Publisher flow control or drop           │
└────────────────────────────────────────────────────────────┘
```

---

## 3.2.2 State 1: Publishing

```python
# rabbitmq_service.py — publish_message()
self._channel.basic_publish(
    exchange='',
    routing_key='rag_processing_queue',
    body=message_body,
    properties=pika.BasicProperties(
        delivery_mode=2,        # Mark as persistent (written to disk)
        content_type='application/json',
    )
)
```

**What happens internally:**
1. pika serializes the AMQP frame
2. Sends bytes over TCP to RabbitMQ broker
3. Broker receives and routes to exchange
4. Exchange routes to matching queue
5. `basic_publish` returns (fire-and-forget by default)

**delivery_mode=2:** Tells RabbitMQ to write the message to disk before confirming. Without this, messages live only in memory and are lost on broker restart.

---

## 3.2.3 State 2: In Exchange — Routing

The exchange examines the routing key and forwards to bound queues in microseconds. If no queue matches the routing key and the exchange is not set to `mandatory=True`, the message is **silently dropped**.

```python
# With mandatory=True — publisher gets an error if no queue matches
channel.basic_publish(
    exchange='my_exchange',
    routing_key='unmatched.key',
    body=b'important message',
    mandatory=True  # Returns unroutable message to publisher callback
)
```

**Your system:** No `mandatory=True` — if the queue doesn't exist, the message is dropped silently. In practice, the queue is always declared on startup.

---

## 3.2.4 State 3: Queued — Waiting for Consumer

Once in the queue, the message waits. Key attributes in this state:

```
Queue: rag_processing_queue
Properties set at queue declaration (rabbitmq_service.py):
    durable=True          → Queue survives broker restart
    
Message properties (from publish):
    delivery_mode=2       → Message body survives broker restart
    
Both must be true for full persistence.
Without durable=True queue:   Queue lost on restart, messages lost too.
Without delivery_mode=2:      Queue exists on restart but messages gone.
```

**Worker prefetch:**
```python
# chat_worker.py — limits how many unacked messages worker holds at once
channel.basic_qos(prefetch_count=1)
```
With `prefetch_count=1`, RabbitMQ delivers at most 1 message to the worker at a time. The next message is not sent until the first is acknowledged. This prevents a fast worker from hoarding messages while slow workers sit idle.

---

## 3.2.5 State 4: Delivered — Unacknowledged

When the consumer receives a message, it enters the **Unacknowledged** state. The message is still tracked by the broker in an "unacked" list:

```
Queue visible: 0 messages (they are with consumers)
Unacked list:  1 message (job_id: abc, consumer: worker_1)
```

If the consumer disconnects while a message is Unacknowledged, the broker **automatically requeues** it:

```
Worker dies mid-processing
    ↓
TCP connection closes
    ↓
Broker detects disconnection
    ↓
Moves message from Unacked back to Ready in queue
    ↓
Delivered to next available consumer
```

This is the core reliability guarantee: **at-least-once delivery**.

---

## 3.2.6 State 5a: Acknowledged (ACK) — Success

```python
# In chat_worker.py _on_message callback
channel.basic_ack(delivery_tag=method.delivery_tag)
```

The broker removes the message permanently from the queue and unacked list. The message is gone — no recovery possible.

**Your system's ACK strategy:**
```
Message received
    ↓
Process RAG query (may take 10+ seconds)
    ↓
Publish result to Redis
    ↓
basic_ack()   ← Only ack AFTER successful processing
```

This is correct — ACK after work ensures no messages are lost if the worker crashes mid-processing.

---

## 3.2.7 State 5b: Negative Acknowledge (NACK) — Failure

```python
channel.basic_nack(
    delivery_tag=method.delivery_tag,
    requeue=False   # Don't put back in queue — discard or DLX
)
```

With `requeue=True`: Message returns to queue head for retry.
With `requeue=False`: Message is discarded or sent to Dead Letter Exchange.

**Your system's NACK strategy (chat_worker.py):**
```python
try:
    result = self._process_job(job_data)
    channel.basic_ack(delivery_tag=tag)
except Exception as e:
    # Publish error response so WebSocket client gets feedback
    self._publish_response(job_id, session_id, error_message)
    channel.basic_nack(delivery_tag=tag, requeue=False)
```

The worker always publishes something to Redis (success or error), then acks or nacks. This ensures the WebSocket client never hangs indefinitely.

## 3.2.8 Publisher Confirms (Not Used, But Important)

By default, `basic_publish` is fire-and-forget — no confirmation from broker. If the broker crashes after you call `basic_publish` but before it writes to disk, the message is lost.

**Publisher Confirms** are the solution:
```python
channel.confirm_delivery()          # Enable confirms on this channel

result = channel.basic_publish(...)
# Returns True if broker confirmed, False if rejected
```

**Your system:** Does not currently use publisher confirms. The 1-retry logic in `publish_message()` handles a stale connection, but not a broker crash mid-confirm. For production, enabling publisher confirms provides stronger delivery guarantees.

## 3.2.9 Interview Questions and Answers

**Q: What happens to an unacknowledged message if the consumer crashes?**

A: When the consumer's TCP connection drops, the broker detects the disconnection and automatically requeues the unacknowledged message. It goes back to Ready status in the queue and will be redelivered to the next available consumer. This is called at-least-once delivery — the message is guaranteed to be processed at least once, but may be processed more than once if the consumer crashes after processing but before acknowledging.

**Q: What is `delivery_mode=2` and when would you omit it?**

A: `delivery_mode=2` marks a message as persistent — the broker writes it to disk before routing. Without it (delivery_mode=1), messages exist only in memory and are lost on broker restart. You might omit persistence for high-throughput, low-latency scenarios where occasional message loss is acceptable (e.g., live analytics counters, ephemeral notifications). For your system's RAG queries, persistence ensures a user's question is not lost even if RabbitMQ restarts mid-queue.
