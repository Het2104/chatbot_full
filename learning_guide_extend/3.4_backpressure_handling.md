## 3.4.1 What Is Backpressure?

Backpressure is the mechanism by which a slow consumer signals to the producer to slow down or stop producing. Without it, fast producers overwhelm slow consumers, causing:
- Queue depth growing unboundedly → memory exhaustion
- Messages timing out waiting for processing
- System instability under load spikes

```
PRODUCER (fast):  [job][job][job][job][job][job]... → 1000 jobs/minute
CONSUMER (slow):  [processing...10 seconds each...] → 6 jobs/minute
                  
Without backpressure: Queue grows 994 jobs/minute → infinite growth
With backpressure:    Producer slows to match consumer capacity
```

## 3.4.2 RabbitMQ's Native Backpressure Mechanisms

### 1. Queue Length Limit

```python
channel.queue_declare(
    queue='rag_processing_queue',
    durable=True,
    arguments={
        'x-max-length': 500,            # Maximum 500 messages in queue
        'x-overflow': 'reject-publish', # Reject new publishes when full
        # OR 'x-overflow': 'drop-head'  # Drop oldest message to make room
    }
)
```

When the queue hits 500 and `reject-publish` is set:
- Publisher's `basic_publish` raises `UnroutableError`
- Application catches it and returns 503 to the user
- Queue does not overflow

### 2. RabbitMQ Memory-Based Flow Control

RabbitMQ has a built-in mechanism where it blocks publishers when memory usage exceeds a threshold (default 40% of available RAM):

```
RabbitMQ memory > 40% of RAM:
    → Sends AMQP flow control frame to all publishers
    → pika's blocking_connection.process_data_events() blocks
    → Publishers are paused until memory drops
```

This is why your `rabbitmq_service.publish_message()` might occasionally take much longer than usual — you're experiencing flow control.

## 3.4.3 Application-Level Backpressure

Check queue depth before publishing:

```python
# rabbitmq_service.py — add queue depth check
def get_queue_depth(self) -> int:
    """Returns current number of messages waiting in queue."""
    if not self.is_available():
        return -1
    
    method = self._channel.queue_declare(
        queue='rag_processing_queue',
        passive=True    # Don't create — just check
    )
    return method.method.message_count

def publish_with_backpressure(self, message: dict, max_depth: int = 200) -> bool:
    """Returns False and rejects the request if queue is too deep."""
    depth = self.get_queue_depth()
    if depth > max_depth:
        logger.warning(f"Queue depth {depth} exceeds limit {max_depth}")
        return False  # Caller returns 503 to user
    
    self.publish_message(message)
    return True
```

```python
# chat router — use backpressure check
@router.post("/chat/message/queue")
async def queue_message(request: ChatRequest, ...):
    
    if not rabbitmq_service.publish_with_backpressure(job_data, max_depth=200):
        raise HTTPException(
            status_code=503,
            detail="System is at capacity. Please try again in a few minutes.",
            headers={"Retry-After": "60"}
        )
    
    return {"job_id": job_id, "status": "queued"}
```

## 3.4.4 Consumer-Side Throttling with prefetch_count

```python
# chat_worker.py
channel.basic_qos(prefetch_count=1)
```

`prefetch_count=N` tells RabbitMQ: "deliver at most N messages to this consumer before waiting for ACKs."

```
Without prefetch_count:
    Worker → ACK message 1 → immediately gets messages 2, 3, 4, 5...
    Worker memory fills up with queued but unprocessed messages
    If worker crashes, all those unacked messages must be redelivered

prefetch_count=1:
    Worker → ACK message 1 → gets message 2 → ACK message 2 → gets 3...
    At most 1 message "in flight" per worker
    Enables fair dispatch across multiple workers (no single worker hogs messages)
```

**Fair dispatch with multiple workers:**
```
Queue: [Job1][Job2][Job3][Job4][Job5]

Worker A (fast server): prefetch=1
    Gets Job1, ACKs → Gets Job3, ACKs → Gets Job5, ACKs...

Worker B (slow server): prefetch=1  
    Gets Job2, (still processing) → RabbitMQ waits to give Job4
    Only gets Job4 after ACKing Job2

Result: Fast worker A does more work (fair capacity-based dispatch)
```

**Without prefetch:** Worker A could grab Jobs 1, 3, 5 instantly while Worker B only has Job 2 — load is not fairly distributed.

## 3.4.5 Monitoring Queue Depth as a Signal

Queue depth is a leading indicator of system health:

```
Queue depth = 0:      Workers keeping up — good
Queue depth = 10:     Normal backlog — fine
Queue depth = 100:    Workers falling behind — investigate
Queue depth = 500:    Workers severely overloaded — add workers or reduce load
Queue depth = 5000:   Critical — system degrading, jobs expiring
```

**Alert thresholds:**
```
Slack alert:    depth > 50  for 5 minutes
Pager alert:    depth > 200 for 2 minutes
Auto-scale:     depth > 100 → spin up additional worker container
```

**RabbitMQ Management API for monitoring:**
```python
import requests

def get_queue_stats(rabbitmq_host: str, queue_name: str) -> dict:
    response = requests.get(
        f"http://{rabbitmq_host}:15672/api/queues/%2F/{queue_name}",
        auth=("guest", "guest")
    )
    data = response.json()
    return {
        "messages": data["messages"],               # Total in queue
        "consumers": data["consumers"],             # Active consumers
        "messages_ready": data["messages_ready"],   # Ready to deliver
        "messages_unacked": data["messages_unacknowledged"],  # In-flight
        "publish_rate": data["message_stats"]["publish_details"]["rate"],
        "consume_rate": data["message_stats"]["deliver_get_details"]["rate"],
    }
```

## 3.4.6 Consumer Scaling Based on Queue Depth

In Docker Compose/Kubernetes, scale workers based on queue depth:

```python
# auto_scaler.py — example scaling logic
def check_and_scale():
    stats = get_queue_stats("rabbitmq", "rag_processing_queue")
    depth = stats["messages"]
    current_workers = stats["consumers"]
    
    desired_workers = max(1, min(10, depth // 50))  # 1 worker per 50 queued jobs
    
    if desired_workers > current_workers:
        logger.info(f"Scaling UP: {current_workers} → {desired_workers} workers")
        scale_service("chat_worker", desired_workers)
    elif desired_workers < current_workers - 1:  # Hysteresis to prevent flapping
        logger.info(f"Scaling DOWN: {current_workers} → {desired_workers} workers")
        scale_service("chat_worker", desired_workers)
```

## 3.4.7 Interview Questions and Answers

**Q: What is backpressure and why is it important in your system?**

A: Backpressure is a flow control mechanism that prevents fast producers from overwhelming slow consumers. In your system, if 500 users submit RAG queries simultaneously but workers can only process 6 per minute, without backpressure the queue would grow to thousands of messages. Users who submitted early would wait 30+ minutes for responses (or timeout), while new users kept submitting. With backpressure (queue depth limit + 503 response), the system tells users "at capacity" immediately, maintains reasonable queue depths, and ensures users who are accepted get prompt responses.

**Q: What does `prefetch_count=1` do and why is it used?**

A: `prefetch_count=1` limits each worker to holding at most 1 unacknowledged message at a time. RabbitMQ only sends the next message after the current one is ACK'd. Two benefits: 1) Fair dispatch — multiple workers get jobs proportional to their processing speed (fast workers get more by ACKing faster). 2) Reliability — if a worker crashes, at most 1 in-flight message needs redelivery. The tradeoff is slightly reduced throughput per worker compared to prefetch=10, but for 10-second RAG jobs the difference is negligible.
