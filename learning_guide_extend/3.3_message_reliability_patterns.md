## 3.3.1 The At-Least-Once vs Exactly-Once Spectrum

```
AT-MOST-ONCE:
    Message may be lost. Never duplicated.
    Use when: Loss is acceptable (metrics, analytics, logs)
    RabbitMQ config: auto-ack, delivery_mode=1 (non-persistent)

AT-LEAST-ONCE:
    Message guaranteed to be delivered. May be duplicated.
    Use when: Loss is unacceptable. Consumer must handle duplicates.
    RabbitMQ config: manual ack, delivery_mode=2 (persistent)
    YOUR SYSTEM uses this.

EXACTLY-ONCE:
    Message delivered exactly one time. Very expensive.
    Use when: Absolute requirement (financial transactions)
    Implementation: Idempotency keys + deduplication on consumer
    RabbitMQ: Cannot guarantee natively — needs application-level dedup
```

## 3.3.2 Dead Letter Exchange (DLX) — Failed Message Routing

When a message cannot be processed, it should go somewhere for inspection rather than vanish:

```python
# queue declaration with DLX
channel.queue_declare(
    queue='rag_processing_queue',
    durable=True,
    arguments={
        'x-dead-letter-exchange': 'dlx',           # Route failed messages here
        'x-dead-letter-routing-key': 'failed.rag'  # With this routing key
    }
)

# Declare the dead letter exchange and queue
channel.exchange_declare(exchange='dlx', exchange_type='direct', durable=True)
channel.queue_declare(queue='rag_failed_queue', durable=True)
channel.queue_bind(exchange='dlx', queue='rag_failed_queue', routing_key='failed.rag')
```

**Messages move to DLX when:**
1. Consumer calls `basic_nack(requeue=False)`
2. Message TTL expires (x-message-ttl)
3. Queue length limit exceeded (x-max-length)

**Your current system:** No DLX. Failed jobs are NACK'd and discarded. For production, messages in the dead letter queue could be:
- Inspected manually
- Retried after human review
- Trigger alerts for operations teams

## 3.3.3 Retry Queue Pattern

Instead of immediately NACK'ing with `requeue=False`, implement delayed retry:

```
Normal flow:
    rag_queue → Worker → Success → ACK, done

Failure flow (with retries):
    rag_queue → Worker → Failure (attempt 1)
        → NACK → retry_queue (with 30s TTL delay)
        → (30s passes, TTL expires) → DLX → rag_queue (requeued)
        → Worker → Failure (attempt 2)
        → NACK → retry_queue (with 60s TTL delay)
        → ... up to max_retries
        → max reached → dead_letter_queue for human inspection
```

**Implementation using message TTL on retry queue:**
```python
# Retry queue with 30-second TTL
channel.queue_declare(
    queue='rag_retry_queue',
    durable=True,
    arguments={
        'x-message-ttl': 30000,              # Messages expire after 30s
        'x-dead-letter-exchange': '',        # Route expired messages to...
        'x-dead-letter-routing-key': 'rag_processing_queue'  # ...back to main queue
    }
)
```

**In the worker:**
```python
def _on_message(self, channel, method, properties, body):
    retry_count = properties.headers.get('x-retry-count', 0)
    
    try:
        self._process_job(body)
        channel.basic_ack(method.delivery_tag)
    except RetryableError:
        if retry_count < MAX_RETRIES:
            # Send to retry queue with incremented count
            headers = {'x-retry-count': retry_count + 1}
            channel.basic_publish(
                exchange='',
                routing_key='rag_retry_queue',
                body=body,
                properties=pika.BasicProperties(
                    delivery_mode=2,
                    headers=headers
                )
            )
        else:
            # Max retries exceeded — send to DLQ
            channel.basic_nack(method.delivery_tag, requeue=False)
        channel.basic_ack(method.delivery_tag)  # Ack original
    except NonRetryableError:
        channel.basic_nack(method.delivery_tag, requeue=False)
```

## 3.3.4 Publisher Confirms

By default, `basic_publish` returns immediately without confirmation the broker received the message. If the broker crashes between publish and disk write, the message is lost.

**Publisher Confirms** provide acknowledgment from the broker:
```python
channel.confirm_delivery()  # Enable publisher confirms on this channel

# After this, basic_publish blocks until broker acks or nacks
try:
    channel.basic_publish(
        exchange='',
        routing_key='rag_processing_queue',
        body=message,
        mandatory=True  # Get error if message can't be routed
    )
except pika.exceptions.UnroutableError:
    logger.error("Message unroutable — queue may not exist")
```

**Tradeoff:** Publisher confirms add ~1-5ms latency per message (broker must write to disk first). For your system's workload (one message per user query), this is acceptable and recommended.

## 3.3.5 Message TTL (Time-To-Live)

Messages sitting unprocessed for too long may become irrelevant:

```python
# Per-message TTL (overrides queue TTL if lower)
channel.basic_publish(
    exchange='',
    routing_key='rag_processing_queue',
    body=message,
    properties=pika.BasicProperties(
        delivery_mode=2,
        expiration='300000'  # 5 minutes in milliseconds (string!)
    )
)
# If worker doesn't pick this up within 5 minutes → DLX or discarded
```

**Queue-level TTL:**
```python
channel.queue_declare(
    queue='rag_processing_queue',
    durable=True,
    arguments={'x-message-ttl': 300000}  # All messages expire after 5 min
)
```

**Why:** If your workers are down for 10 minutes and 1000 jobs accumulated, when workers restart they'd process all 1000 — but the WebSocket clients that submitted them have likely timed out and disconnected. A 5-minute TTL discards these stale jobs automatically.

## 3.3.6 Message Deduplication for Exactly-Once

```python
# Add unique job ID to message properties
channel.basic_publish(
    exchange='',
    routing_key='rag_processing_queue',
    body=message,
    properties=pika.BasicProperties(
        delivery_mode=2,
        message_id=job_id     # Unique ID for deduplication
    )
)

# Worker checks if already processed
def _process_job(self, channel, method, properties, body):
    job_id = properties.message_id
    
    # Check Redis for deduplication
    if redis_client.exists(f"processed:{job_id}"):
        logger.info(f"Duplicate job {job_id} detected, skipping")
        channel.basic_ack(method.delivery_tag)  # Ack without reprocessing
        return
    
    # Process the job
    result = run_rag_pipeline(body)
    
    # Mark as processed (TTL = 24 hours, longer than retry window)
    redis_client.set(f"processed:{job_id}", "1", ex=86400)
    
    channel.basic_ack(method.delivery_tag)
```

## 3.3.7 Interview Questions and Answers

**Q: What is at-least-once delivery and how does your system handle duplicates?**

A: At-least-once delivery means RabbitMQ guarantees a message is processed at least once, but may deliver it more than once if the worker ACKs after processing but before the confirmation reaches the broker (crash between these steps). Your system handles this acceptably for RAG queries: a duplicate job would publish a duplicate response to Redis, but the WebSocket handler reads the first response and the second expires with the Redis TTL. For a payment system, you'd need idempotency keys in a database to detect and skip duplicates.

**Q: Why would you use a retry queue with TTL instead of `basic_nack(requeue=True)`?**

A: `requeue=True` puts the message back at the HEAD of the queue and it's immediately redelivered — if the error is caused by a temporary downstream issue (e.g., Milvus starting up), you'd create a tight retry loop consuming CPU. A TTL-based retry queue delays the redelivery, giving the downstream service time to recover, and limits retries through an x-retry-count header. This is the exponential backoff pattern applied to message queuing.
