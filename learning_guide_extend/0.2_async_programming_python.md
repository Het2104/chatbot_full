## 0.2.1 Why Async Matters in This System

Your FastAPI server must handle many concurrent users simultaneously. Without async programming, a single slow operation (like waiting for OpenAI — 10 seconds) would block the server from serving ANY other request during that time.

Async programming lets the server switch between tasks while any one task is waiting for I/O (network, disk). This is the foundation of FastAPI's performance.

## 0.2.2 Synchronous vs Asynchronous — Core Concept

### Synchronous (blocking):

```python
import requests
import time

def handle_user_a():
    result = requests.get("https://api.openai.com/...")  # BLOCKS for 10 seconds
    return result                                         # Nobody else served during this

def handle_user_b():
    # This waits for user_a to finish before starting
    pass
```

**Timeline:**
```
Thread 1: [User A → Wait 10s → Done][User B → Wait 10s → Done][User C → ...]
      Time: 0         10        20         30
```

### Asynchronous (non-blocking):

```python
import httpx
import asyncio

async def handle_user_a():
    result = await httpx.AsyncClient().get("https://api.openai.com/...")
    # While waiting, give control back to event loop → serve other users
    return result

async def handle_user_b():
    # Can start even while user_a is waiting
    pass
```

**Timeline:**
```
Event Loop: [User A starts][User B starts][User C starts][A returns][B returns][C returns]
       Time: 0              0.1            0.2            10          10.1        10.2
```

All three users' AI calls run "simultaneously" (interleaved, not parallel) and all finish around t=10.

## 0.2.3 The `async` / `await` Keywords

```python
# Declare an async function using `async def`
async def my_async_function():
    # `await` suspends this coroutine and returns control to event loop
    result = await some_async_operation()
    return result
```

**Rules:**
- You can only `await` inside an `async def` function
- `await` can only be used with **coroutines** or **awaitables** (objects with `__await__`)
- Calling `async def` does NOT run it — it creates a coroutine object. You must `await` it or schedule it.

```python
# WRONG — this creates a coroutine object but doesn't run it
result = my_async_function()

# CORRECT — this runs it
result = await my_async_function()
```

## 0.2.4 FastAPI and Async Routes

FastAPI supports both sync and async route handlers:

```python
# Async route (preferred for I/O-bound operations)
@router.post("/chat/message/queue")
async def queue_message(request: ChatRequest, ...):
    # Can await other async functions
    result = await some_async_db_query()
    return result

# Sync route (FastAPI runs this in a thread pool automatically)
@router.get("/health")
def health_check():
    # Still works, FastAPI wraps it
    return {"status": "ok"}
```

**FastAPI's behavior with sync routes:** It wraps them in `asyncio.run_in_executor()` automatically, so they don't block the event loop. But async routes are more efficient for I/O-heavy workloads.

## 0.2.5 `asyncio.to_thread()` — The Bridge

Your WebSocket handler needs to call `RedisService.listen_once()` — a **blocking** function (it uses `pubsub.get_message()` which blocks waiting for bytes).

You cannot call blocking code directly inside an async function. The solution:

```python
# In routers/websocket.py
result = await asyncio.to_thread(
    pubsub_service.collect_job_response,
    job_id,
    WEBSOCKET_RESPONSE_TIMEOUT,
)
```

`asyncio.to_thread()` runs the blocking function in a **thread pool executor**, separate from the event loop. The event loop is freed to handle other requests while the Redis blocking call runs in its own thread. When the blocking call returns, the result is delivered back to the async function.

## 0.2.6 When to Use Async vs Sync

| Operation | Use | Reason |
|---|---|---|
| Database query (SQLAlchemy sync) | `asyncio.to_thread()` or sync route | SQLAlchemy sync ORM is blocking |
| HTTP call (httpx.AsyncClient) | `async def` + `await` | Truly non-blocking |
| Redis (redis-py sync) | `asyncio.to_thread()` or sync route | redis-py blocking client |
| RabbitMQ consume (pika) | Background thread | pika blocks the thread |
| File read | `asyncio.to_thread()` | `open()` is blocking |
| CPU computation | Thread or process pool | Don't block event loop with CPU work |

**In your system:**
- `collect_job_response()` is blocking → called via `asyncio.to_thread()`
- Database operations in routes → sync SQLAlchemy in sync dependency
- RabbitMQ consume loop → background thread (`chat_worker.start(run_in_thread=True)`)

## 0.2.7 Coroutines, Tasks, and Futures

```
Coroutine:  An async function's return value. Not running yet.
            result = my_fn()  → coroutine object

Task:       A coroutine scheduled onto the event loop to run concurrently.
            task = asyncio.create_task(my_fn())  → Task (running)

Future:     A placeholder for a value not yet available.
            asyncio.to_thread() returns a Future
```

**Creating concurrent tasks:**
```python
async def main():
    # These run concurrently — not sequentially
    task_a = asyncio.create_task(fetch_user())
    task_b = asyncio.create_task(fetch_profile())
    
    user, profile = await asyncio.gather(task_a, task_b)
```

## 0.2.8 async/await vs Threads — When to Use Each

| | async/await | Threads |
|---|---|---|
| **Best for** | I/O-bound (network, disk) | CPU-bound or blocking libraries |
| **Overhead** | Very low (no OS context switch) | Higher (OS thread context switch) |
| **Safety** | Single-threaded: no race conditions | Multi-threaded: need locks |
| **In your system** | FastAPI routes, WebSocket handlers | ChatWorker consumer loop, blocking Redis calls |

## 0.2.9 Interview Questions and Answers

**Q: What is the difference between synchronous and asynchronous code in Python?**

A: Synchronous code executes sequentially — each operation waits for the previous one to complete. Asynchronous code uses `async/await` with an event loop to suspend operations that are waiting for I/O, allowing other operations to run. This enables handling many concurrent requests without needing a thread per request.

**Q: What does `await` actually do?**

A: `await` suspends the current coroutine, returns control to the event loop, and registers a callback to resume the coroutine when the awaited operation completes. The event loop is then free to run other coroutines. No CPU time is wasted waiting.

**Q: Why do you need `asyncio.to_thread()` for Redis operations?**

A: The `redis-py` client's `get_message()` call is a blocking operation — it waits for bytes from the network using a blocking socket. Calling it directly inside an async function would freeze the event loop because Python's async is single-threaded. `asyncio.to_thread()` offloads the blocking call to a thread pool while keeping the event loop free.
