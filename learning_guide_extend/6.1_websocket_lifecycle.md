## 6.1.1 The WebSocket State Machine

A WebSocket connection moves through distinct states. Understanding each is required for correct cleanup code.

```
STATES:
    CONNECTING (0) → OPEN (1) → CLOSING (2) → CLOSED (3)

Transitions:
    CONNECTING → OPEN:    Handshake succeeded
    OPEN → CLOSING:       Either party initiates close (sends Close frame)
    CLOSING → CLOSED:     Both parties exchanged Close frames
    CONNECTING → CLOSED:  Handshake failed
    OPEN → CLOSED:        Abrupt disconnect (TCP reset, no Close frame)
```

## 6.1.2 The HTTP Upgrade Handshake (Revisited in Detail)

```
Browser                         FastAPI Server
    │                                 │
    │ GET /ws/chat/{session_id}        │
    │ Connection: Upgrade             │
    │ Upgrade: websocket              │
    │ Sec-WebSocket-Key: dGhlc2FtcGxl │  (random base64)
    │ Sec-WebSocket-Version: 13       │
    ├────────────────────────────────→│
    │                                 │ Compute accept key:
    │                                 │ SHA1(key + MAGIC_GUID)
    │                                 │ Base64 encode result
    │                                 │
    │ HTTP/1.1 101 Switching Protocols│
    │ Connection: Upgrade             │
    │ Upgrade: websocket              │
    │ Sec-WebSocket-Accept: s3pPLMBi  │
    │←────────────────────────────────│
    │                                 │
    │ [WebSocket frames from here on] │
```

After the 101 response, the TCP connection is no longer HTTP. Both sides communicate using the WebSocket framing protocol.

## 6.1.3 WebSocket Frame Types

Beyond text/binary data, WebSocket defines control frames:

| Frame Type | Opcode | Description |
|---|---|---|
| Text | 0x1 | UTF-8 text message |
| Binary | 0x2 | Raw binary message |
| Close | 0x8 | Initiate closing handshake |
| Ping | 0x9 | Keepalive probe |
| Pong | 0xA | Response to Ping |
| Continuation | 0x0 | Fragment of a previous message |

**Ping/Pong keepalive:**
```
Server → Ping frame (every 30 seconds) → Client
Server ← Pong frame ←────────────────── Client

If no Pong received within timeout → server closes connection
```

This detects "zombie" connections — browser tabs that were closed without sending a Close frame (e.g., laptop dropped off WiFi).

## 6.1.4 WebSocket Close Codes

The Close frame carries a numeric status code and optional reason:

| Code | Name | Meaning |
|---|---|---|
| 1000 | Normal Closure | Clean close, task complete |
| 1001 | Going Away | Server shutdown or browser navigated away |
| 1002 | Protocol Error | WebSocket protocol violation |
| 1003 | Unsupported Data | Received data type not supported |
| 1006 | Abnormal Closure | TCP terminated without close frame (network failure) |
| 1007 | Invalid Frame Payload | Non-UTF8 in text frame |
| 1008 | Policy Violation | Server-specific rule violated (e.g., auth failed) |
| 1009 | Message Too Big | Message exceeds configured limit |
| 1011 | Internal Error | Unexpected server error |

**In your WebSocket handler**, FastAPI/Starlette handles close codes automatically. The `WebSocketDisconnect` exception is raised when the client disconnects, and the code is available in `exc.code`.

## 6.1.5 WebSocket Handler Lifecycle in Your System

```python
# backend/app/routers/websocket.py
@router.websocket("/ws/chat/{session_id}")
async def websocket_chat(websocket: WebSocket, session_id: str, ...):
    
    # 1. ACCEPT — Completes the HTTP→WS upgrade
    await websocket.accept()
    
    # 2. RECEIVE INITIAL MESSAGE
    data = await websocket.receive_text()
    request = json.loads(data)
    
    # 3. VALIDATE — Auth, input validation
    
    # 4. PUBLISH JOB TO RABBITMQ
    await rabbitmq_service.publish_message(job_data)
    
    # 5. WAIT FOR RESULT (blocks in thread)
    result = await asyncio.to_thread(
        pubsub_service.collect_job_response, job_id, timeout
    )
    
    # 6. SEND RESPONSE
    await websocket.send_text(json.dumps({"result": result}))
    
    # 7. CLOSE
    await websocket.close()
    # Handler returns → FastAPI cleans up

# If client disconnects at any point during steps 2-7:
#     → websocket.receive_text() or send_text() raises WebSocketDisconnect
#     → Handler exits (finally block runs if any)
```

## 6.1.6 Disconnect Detection

```python
# Explicit disconnect handling pattern
@router.websocket("/ws/example")
async def ws_handler(websocket: WebSocket):
    await websocket.accept()
    
    try:
        while True:
            data = await websocket.receive_text()   # Raises WebSocketDisconnect if client gone
            response = process(data)
            await websocket.send_text(response)     # Also raises on disconnect
    
    except WebSocketDisconnect as e:
        print(f"Client disconnected with code {e.code}")
        # Cleanup: unsubscribe from channels, cancel pending jobs, etc.
    
    finally:
        # Cleanup that ALWAYS runs, disconnect or normal close
        await websocket.close()  # safe to call even if already closed
```

**In your system's single-request model**, the handler doesn't loop — it processes one message and closes. The WebSocket is long-lived only during the AI processing wait (up to 60+ seconds). If the client closes the browser tab during this wait:
1. TCP FIN or RST packet received
2. `asyncio.to_thread()` is still running (in thread pool)
3. Thread eventually returns (Redis timeout)
4. Next `send_text()` raises WebSocketDisconnect
5. Handler exits, resources cleaned up

## 6.1.7 Heartbeat / Keepalive

Long-lived WebSocket connections can be silently dropped by:
- Load balancers (idle connection timeout, typically 60 seconds)
- NAT gateways (idle flow expired)
- Mobile networks (IP change / connection reset)

**Solution: Application-level heartbeat**
```python
async def ws_handler_with_heartbeat(websocket: WebSocket):
    await websocket.accept()
    
    heartbeat_task = asyncio.create_task(send_heartbeats(websocket))
    
    try:
        data = await websocket.receive_text()
        # ... process
        await websocket.send_text(result)
    finally:
        heartbeat_task.cancel()
        await websocket.close()

async def send_heartbeats(websocket: WebSocket):
    while True:
        await asyncio.sleep(30)
        try:
            await websocket.send_text(json.dumps({"type": "ping"}))
        except WebSocketDisconnect:
            break
```

**Your system:** Does not currently implement heartbeats. For production, especially behind load balancers, you'd want either: application-level pings or configuring the load balancer's idle timeout to exceed your max response time.

## 6.1.9 Interview Questions and Answers

**Q: What happens if the user closes their browser tab while waiting for the AI response?**

A: The browser sends a TCP FIN packet (or RST in some cases), closing the underlying socket. The `collect_job_response()` function is still running in a thread pool — it continues waiting for the Redis response. When it eventually returns (success or timeout), the WebSocket handler tries to send the result with `websocket.send_text()`. This raises a `WebSocketDisconnect` exception since the connection is closed. The exception propagates, the handler exits, and all resources are cleaned up. The RAG processing completes normally in the background — no resource leak, but the result is discarded.

**Q: What are WebSocket close codes and when would you use 1008?**

A: Close codes are numeric status codes sent in the Close frame to indicate why the connection is being closed. Code 1008 (Policy Violation) would be used when a server-side rule is violated — for example, if a user sends a malformed authentication message at the start of the WebSocket connection, you could close with 1008 to indicate "rejected due to policy/auth failure." This helps the client understand why the connection was closed, rather than seeing a generic 1006 "abnormal closure."
