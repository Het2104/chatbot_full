## 2.1.1 Defense in Depth

No single security control is sufficient. Layer multiple defenses so a failure in one layer is caught by another:

```
Layer 1: Network         — Firewall, private subnets, TLS
Layer 2: Authentication  — JWT, bcrypt, session management
Layer 3: Authorization   — Role checks on every endpoint
Layer 4: Input Validation — Pydantic, SQL parameterization
Layer 5: Secrets Mgmt    — Environment variables, never in code
Layer 6: Monitoring      — Log auth failures, detect anomalies
Layer 7: Dependency Mgmt — Updated libraries, vulnerability scanning
```

## 2.1.2 Secrets Management

Secrets (API keys, DB passwords, JWT secret) must NEVER appear in source code or version control.

**Bad (never do this):**
```python
# hardcoded in source — exposed via git history forever
JWT_SECRET = "my-secret-key-12345"
OPENAI_API_KEY = "sk-abc123realkey"
DATABASE_URL = "postgresql://user:realpassword@localhost/db"
```

**Good:**
```python
# config.py — read from environment variables
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    jwt_secret_key: str          # Must be set in env
    openai_api_key: str          # Must be set in env
    database_url: str            # Must be set in env
    
    class Config:
        env_file = ".env"        # Local dev only — .env in .gitignore
```

**Docker Compose secrets:**
```yaml
# docker-compose.yml
services:
  app:
    env_file:
      - .env          # Loaded from .env file at runtime
    environment:
      - JWT_SECRET_KEY=${JWT_SECRET_KEY}  # From host environment
```

**Production secrets management:**
- AWS Secrets Manager / Parameter Store
- HashiCorp Vault
- Kubernetes Secrets (base64 encoded, namespace isolated)

## 2.1.3 Password Security

```python
# auth_service.py — bcrypt hashing
from passlib.context import CryptContext

pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

def hash_password(password: str) -> str:
    return pwd_context.hash(password)
    # Returns: "$2b$12$Gu8BsYFJdqQBP1w3qP7..."
    # The "$12$" is the work factor (2^12 = 4096 hash rounds)

def verify_password(plain: str, hashed: str) -> bool:
    return pwd_context.verify(plain, hashed)
    # Timing-safe comparison — prevents timing attacks
```

**Why bcrypt:**
- Intentionally slow (tunable work factor) — makes brute force expensive
- Built-in salt — prevents rainbow table attacks
- One-way — cannot be reversed (even by you)

**Password policy:**
```python
class PasswordValidator:
    @staticmethod
    def validate(password: str):
        errors = []
        if len(password) < 8:
            errors.append("Minimum 8 characters")
        if not re.search(r'[A-Z]', password):
            errors.append("Requires uppercase letter")
        if not re.search(r'[0-9]', password):
            errors.append("Requires number")
        if errors:
            raise ValueError(errors)
```

## 2.1.4 JWT Security

```python
# auth_service.py
SECRET_KEY = settings.jwt_secret_key    # Must be long, random, secret
ALGORITHM = "HS256"
ACCESS_TOKEN_EXPIRE_MINUTES = 30         # Short-lived

def create_access_token(data: dict) -> str:
    payload = data.copy()
    payload["exp"] = datetime.utcnow() + timedelta(minutes=ACCESS_TOKEN_EXPIRE_MINUTES)
    payload["iat"] = datetime.utcnow()   # Issued-at timestamp
    return jwt.encode(payload, SECRET_KEY, algorithm=ALGORITHM)
```

**JWT Security risks and mitigations:**

| Risk | Mitigation |
|---|---|
| Token theft (XSS) | HttpOnly cookies (can't be read by JS) OR short expiry + refresh tokens |
| Token theft (MITM) | HTTPS only — never transmit JWT over HTTP |
| `alg: none` attack | Backend explicitly validates algorithm: `algorithms=["HS256"]` |
| Long-lived tokens | Short expiry (15-60 min) + refresh token pattern |
| No revocation | Maintain blacklist in Redis for logout |

**Token revocation (logout):**
```python
# On logout, add JTI (JWT ID) to Redis blacklist
async def logout(token: str):
    payload = decode_access_token(token)
    jti = payload.get("jti")
    ttl = payload["exp"] - time.time()
    redis.set(f"blacklist:{jti}", "1", ex=int(ttl))

# On every request, check blacklist
def decode_access_token(token: str):
    payload = jwt.decode(token, SECRET_KEY, algorithms=["HS256"])
    jti = payload.get("jti")
    if redis.exists(f"blacklist:{jti}"):
        raise HTTPException(status_code=401, detail="Token revoked")
    return payload
```

## 2.1.5 Input Validation

Every user-provided value is hostile until validated. Pydantic enforces this at the schema level:

```python
class ChatMessageRequest(BaseModel):
    message: str = Field(
        ...,
        min_length=1,        # No empty messages
        max_length=4096,     # Prevent huge payloads
        strip_whitespace=True
    )
    session_id: str = Field(
        ...,
        regex=r'^[a-f0-9-]{36}$'  # UUID format only — prevents injection
    )
```

**SQL injection prevention (automatic with SQLAlchemy ORM):**
```python
# SAFE — parameterized query (ORM does this automatically)
user = db.query(User).filter(User.email == email).first()
# Generates: SELECT * FROM users WHERE email = $1; params=('user@example.com',)

# DANGEROUS — string interpolation (never do this)
user = db.execute(f"SELECT * FROM users WHERE email = '{email}'")
# If email = "'; DROP TABLE users;--" — SQL injection!
```

## 2.1.6 HTTPS and TLS

All API communication in production must use TLS:

```nginx
# Nginx TLS termination
server {
    listen 443 ssl;
    ssl_certificate     /etc/nginx/ssl/cert.pem;
    ssl_certificate_key /etc/nginx/ssl/key.pem;
    ssl_protocols       TLSv1.2 TLSv1.3;    # Disable old protocols
    ssl_ciphers         HIGH:!aNULL:!MD5;   # Strong ciphers only
    
    # HSTS — force HTTPS for 1 year
    add_header Strict-Transport-Security "max-age=31536000" always;
    
    location / {
        proxy_pass http://app:8000;  # Internal traffic can be HTTP
    }
}
```

**WebSocket over TLS:** Uses `wss://` (WebSocket Secure) — same TLS terminated by Nginx, then forwarded as `ws://` internally.

## 2.1.7 CORS (Cross-Origin Resource Sharing)

Browsers enforce CORS — a script on `https://myfrontend.com` cannot call `https://myapi.com` without the API explicitly allowing it:

```python
# main.py
from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=["https://myfrontend.com"],  # Specific origin(s)
    # NOT: allow_origins=["*"]               # Never in production with credentials
    allow_credentials=True,                   # Allow cookies/auth headers
    allow_methods=["GET", "POST", "PUT", "DELETE"],
    allow_headers=["Authorization", "Content-Type"],
)
```

**Never use `allow_origins=["*"]` with `allow_credentials=True`** — this is a security vulnerability allowing any website to make credentialed requests to your API.

## 2.1.8 Interview Questions and Answers

**Q: Why is bcrypt used instead of SHA-256 for password hashing?**

A: SHA-256 is designed to be fast — a modern GPU can compute billions of SHA-256 hashes per second, making brute-force attacks trivial. bcrypt is intentionally slow and has a tunable work factor — at factor 12, each hash takes ~250ms, making brute-forcing billions of times slower. SHA-256 is appropriate for cryptographic integrity (checksums, HMAC), not for password storage where slow is better.

**Q: What does HTTPS protect against and what doesn't it protect against?**

A: HTTPS (TLS) protects against: network eavesdropping (attacker reading requests/responses), man-in-the-middle attacks (attacker modifying traffic), replay attacks (reusing captured requests). It does NOT protect against: application vulnerabilities (XSS, SQL injection, broken auth), server-side breaches, compromised endpoints, social engineering. HTTPS secures the transport layer, not the application logic.
