## 11.2.1 Why Structured Logging

A log message is only useful if you can search and filter it. Unstructured logs are hard to machine-parse:

```
# UNSTRUCTURED (bad for search):
2024-01-15 10:23:01 Processing job abc123 for user 42 completed in 8.3 seconds

# STRUCTURED (machine-parseable, filterable):
{
    "timestamp": "2024-01-15T10:23:01.123Z",
    "level": "INFO",
    "service": "chat_worker",
    "event": "job_completed",
    "job_id": "abc123",
    "user_id": 42,
    "duration_seconds": 8.3,
    "environment": "production"
}
```

With structured logs, you can query: "show me all jobs that took > 5 seconds for user 42 in the last hour."

## 11.2.2 Python Logging Architecture

```
Logger (named: "backend.app.worker.chat_worker")
    │
    ├── Handler 1: StreamHandler (console)
    │       └── Formatter: JSON or text
    │
    ├── Handler 2: FileHandler (file rotation)
    │       └── Formatter: JSON
    │
    └── Handler 3: HTTPHandler (log aggregation service)
            └── Sends to: Loki, Elasticsearch, Datadog, etc.

Logger hierarchy:
    root
    └── backend
        └── backend.app
            └── backend.app.worker
                └── backend.app.worker.chat_worker
```

Each logger inherits handlers from its parents unless `propagate=False`.

## 11.2.3 Your System's Logging Configuration

```python
# backend/app/logging_config.py
import logging
import logging.config

LOGGING_CONFIG = {
    "version": 1,
    "disable_existing_loggers": False,
    "formatters": {
        "default": {
            "format": "%(asctime)s - %(name)s - %(levelname)s - %(message)s",
        },
        "json": {
            "()": "pythonjsonlogger.jsonlogger.JsonFormatter",
            "format": "%(timestamp)s %(level)s %(name)s %(message)s",
        },
    },
    "handlers": {
        "console": {
            "class": "logging.StreamHandler",
            "formatter": "default",
            "stream": "ext://sys.stdout",
        },
        "file": {
            "class": "logging.handlers.RotatingFileHandler",
            "formatter": "default",
            "filename": "logs/app.log",
            "maxBytes": 10485760,  # 10 MB
            "backupCount": 5,      # Keep 5 rotated files
        },
    },
    "root": {
        "level": "INFO",
        "handlers": ["console", "file"],
    },
}
```

## 11.2.4 Log Levels — When to Use Each

```
CRITICAL (50) — System is failing, requires immediate human intervention
    logger.critical("Database connection permanently lost, shutting down")
    
ERROR (40) — Operation failed, but system continues
    logger.error("RAG job failed", exc_info=True)  # exc_info adds stack trace
    
WARNING (30) — Unexpected situation, system compensated
    logger.warning("Redis cache miss rate high (>80%), check Redis health")
    
INFO (20) — Normal operations worth recording
    logger.info(f"Job {job_id} completed in {duration:.2f}s")
    
DEBUG (10) — Detailed trace for development
    logger.debug(f"Retrieved {len(chunks)} chunks from Milvus for query")
    
NOTSET (0) — Log everything (almost never use)
```

**Production vs Development:**
- Development: `DEBUG` level on
- Production: `INFO` level — DEBUG is too verbose and can expose sensitive data

## 11.2.5 What to Log in Your System

**Application startup/shutdown:**
```python
logger.info("Application started", extra={
    "rabbitmq": rabbitmq_service.is_available(),
    "redis": redis_cache_service.is_available(),
    "version": "1.0.0"
})
```

**Authentication events (security audit trail):**
```python
logger.info("User login", extra={
    "event": "auth.login.success",
    "user_id": user.id,
    "ip": request.client.host,
})

logger.warning("Failed login attempt", extra={
    "event": "auth.login.failure",
    "username": username,
    "ip": request.client.host,
})
```

**RAG job lifecycle:**
```python
logger.info("RAG job started", extra={
    "job_id": job_id, "session_id": session_id,
    "query_length": len(query), "user_id": user_id
})
logger.info("RAG job completed", extra={
    "job_id": job_id, "duration": elapsed,
    "chunks_retrieved": chunk_count, "response_length": len(response)
})
```

**Service availability changes:**
```python
# WARNING: was available, now not
# INFO: was unavailable, now reconnected
```

## 11.2.6 Log Rotation

Log files grow indefinitely without rotation. Your config uses `RotatingFileHandler`:

```
Rotation happens when file exceeds maxBytes (10 MB)
app.log      ← current log
app.log.1    ← previous (most recent rotation)
app.log.2    ← older
...
app.log.5    ← oldest (backupCount=5)

When app.log reaches 10 MB:
    app.log.5 → deleted
    app.log.4 → app.log.5
    app.log.3 → app.log.4
    app.log.2 → app.log.3
    app.log.1 → app.log.2
    app.log   → app.log.1  (renamed)
    new app.log created
```

Your `logs/` directory shows `app.log.1` — one rotation has occurred.

## 11.2.7 Centralized Log Aggregation

In multi-container deployments, logs from each container need to be collected centrally:

```
Container: app        → stdout → Docker log driver → ┐
Container: worker     → stdout → Docker log driver → ├─→ Loki / ELK / Datadog
Container: rabbitmq   → stdout → Docker log driver → ┘

Query interface: Grafana / Kibana / Datadog UI
```

**Grafana Loki stack (popular for Docker):**
```yaml
# docker-compose.yml addition
logging:
  driver: "loki"
  options:
    loki-url: "http://loki:3100/loki/api/v1/push"
    loki-labels: "service=app"
```

## 11.2.9 Interview Questions and Answers

**Q: What is the advantage of structured (JSON) logging over plain text logging?**

A: Structured logs are machine-parseable — log aggregation tools (Loki, Elasticsearch, Splunk) can index and query by any field without complex regex parsing. You can filter: "show ERROR logs from the chat_worker service in the last hour where duration_seconds > 10." With plain text, you'd need regex to extract fields and manual parsing to filter. Structured logs also make alerting more reliable — you alert on `level=ERROR and service=chat_worker` rather than a fragile regex pattern.

**Q: Why should production logs use INFO level rather than DEBUG?**

A: DEBUG logs contain fine-grained execution details (every SQL query, every Redis key lookup, full request/response bodies) which exposes: 1) sensitive user data in log files, 2) enormous log volume that fills disk and slows log shipping, 3) performance overhead from string formatting millions of debug messages. INFO level captures what happened (job started, job completed, errors) without the implementation details.
