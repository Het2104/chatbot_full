## 1.2.1 The CAP Theorem Defined

The CAP theorem (Brewer's theorem, 2000) states that a distributed data system can guarantee at most **two** of three properties simultaneously:

```
        Consistency (C)
           /\
          /  \
         /    \
        /  CA  \
       /──────  \
      /  CP  AP  \
     /______________\
Availability (A) ── Partition Tolerance (P)
```

**C — Consistency:** Every read receives the most recent write or an error. All nodes see the same data at the same time.

**A — Availability:** Every request receives a response (not an error), but the response might be stale.

**P — Partition Tolerance:** The system continues operating even when network messages are lost or delayed (network partition).

**Critical insight:** Network partitions WILL happen in any distributed system. Therefore P is not really optional — it's a given. The real choice is: when a partition occurs, do you sacrifice C or A?

---

## 1.2.2 CP Systems (Consistency + Partition Tolerance)

Choose to return an error (sacrifice availability) when a partition occurs, rather than return potentially stale data.

```
Example: Your PostgreSQL database
    
    Primary node: has the latest write (user changed password)
    Replica: network partition, not yet synced
    
    User queries the replica:
    CP system → REFUSE the query (return error)
               "Cannot guarantee data is current, refusing to serve"
    
    Result: App gets 503, user sees error
    Benefit: Data is NEVER incorrect
```

Other CP examples: Redis Cluster (when using WAIT), HBase, ZooKeeper, etcd

**When to use CP:** Financial systems, inventory counts, user authentication — anywhere wrong data is worse than no data.

---

## 1.2.3 AP Systems (Availability + Partition Tolerance)

Choose to return possibly stale data rather than refuse the request.

```
Example: Redis FAQ Cache in your system

    Primary Redis: has latest FAQ updates
    Network partition occurs (brief outage)
    App reads from Redis during partition
    
    AP system → RETURN what you have (stale data)
               "I might be stale, but here's an answer"
    
    Result: App gets cached FAQ (possibly 5 minutes old)
    Benefit: System keeps serving users
```

Other AP examples: DynamoDB (default), Cassandra, CouchDB, DNS

**When to use AP:** FAQ answers, product catalog, recommendation systems — anywhere slightly stale data is acceptable.

## 1.2.5 PACELC — The Extended Model

CAP only addresses behavior during partitions. PACELC also asks: what about when there's no partition?

```
P  =  Partition present?
A  →  Availability (choose when partition occurs)
C  →  Consistency (choose when partition occurs)
E  =  Else (no partition)
L  →  Latency (can optimize for lower latency but sacrifice some consistency)
C  →  Consistency (or ensure full consistency but higher latency)
```

Example: PostgreSQL
- During partition: CP (refuse queries to maintain consistency)
- Normal operation: Choose between synchronous replication (EC: consistent but higher latency) or async replication (EL: lower latency but brief inconsistency windows)

## 1.2.6 Consistency Levels (Between Strong and None)

Real systems offer tunable consistency between the extremes:

```
STRONG CONSISTENCY:
    Read is guaranteed to see all previous writes
    Requires synchronization across all replicas
    
MONOTONIC READS:
    You never read older data than you've already seen
    (No time-traveling backward)
    
READ-YOUR-WRITES:
    After you write something, you always read back that value
    (Useful for: user updates their profile and immediately views it)
    
EVENTUAL CONSISTENCY:
    Given no new writes, all replicas eventually converge
    No guarantees on when
```

**FAQ Cache in your system uses "eventual" for most reads but "bypass" for fresh queries:**

```
Write FAQ to DB → Update Redis Cache
Other users may read slightly stale Redis cache (eventual)
But the user who updated immediately bypasses cache (read-your-writes)
```

## 1.2.8 Interview Questions and Answers

**Q: Explain the CAP theorem in simple terms.**

A: CAP theorem says that when a network partition occurs in a distributed system (nodes can't communicate), you must choose: either stop serving requests until the partition heals (choose Consistency, sacrifice Availability), or keep serving requests but potentially return stale data (choose Availability, sacrifice Consistency). Partition tolerance is mandatory in any real distributed system because networks will fail. In practice this means: for critical data like user authentication, choose consistency (better to show an error than incorrect access); for catalog or cache data, choose availability (a slightly stale product price is better than an error page).

**Q: Is your Redis cache CP or AP, and is that the right choice?**

A: The FAQ cache is AP by design and nature — it's a cache, explicitly allowed to hold stale data. Returning a 5-minute-old FAQ answer is vastly better than returning an error because Redis had a partition. For authentication, the system correctly uses PostgreSQL (more consistent) rather than Redis for the authoritative user record. The right choice depends on how harmful stale data is — for FAQs, staleness is harmless; for auth tokens, it could be a security issue.
