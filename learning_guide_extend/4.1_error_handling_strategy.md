## 4.1.1 The Core Contract: Always Publish a Response

The most important error handling principle in your system is the **always-respond contract**:

```
No matter what fails, the WebSocket client MUST receive a message.
A hanging WebSocket is worse than an error response.
```

The `chat_worker.py` enforces this:
```python
def _process_job(self, job_data):
    try:
        result = run_rag_pipeline(query)      # May raise any exception
        self._publish_response(job_id, session_id, result)  # Success
        channel.basic_ack(tag)
    except Exception as e:
        logger.error(f"Job processing failed: {e}")
        self._publish_response(                           # Error response
            job_id, session_id,
            "An error occurred processing your request"
        )
        channel.basic_nack(tag, requeue=False)            # Don't retry
```

The WebSocket handler's timeout also enforces this — if Redis never receives a response within `WEBSOCKET_RESPONSE_TIMEOUT` seconds, the handler returns a timeout error to the client.

## 4.1.2 Layers of Error Handling

Your system handles errors at multiple layers:

```
Layer 1: HTTP Client Errors (FastAPI validation)
    │   Pydantic validates request body → 422 if invalid
    │   JWT decode fails → 401 Unauthorized
    ▼
Layer 2: Service Availability Checks
    │   rabbitmq.is_available() → 503 if RabbitMQ down
    │   redis.is_available() → graceful degradation (cache miss)
    ▼
Layer 3: Queue Processing Errors
    │   Worker try/except → publish error to Redis → NACK
    ▼
Layer 4: Response Timeout
    │   collect_job_response() timeout → return timeout error
    ▼
Layer 5: WebSocket Connection Errors
        disconnect detected → cleanup and exit handler
```

## 4.1.4 Graceful Degradation Pattern

Not all errors should be fatal. Lower-priority services should fail gracefully:

```python
# redis_cache_service.py — graceful degradation on cache failure
async def get(self, key: str):
    try:
        if not self.is_available():
            return None          # No cache → proceed without it
        value = self._client.get(key)
        return value
    except redis.exceptions.ConnectionError:
        logger.warning("Redis cache unavailable, cache miss")
        return None              # Return None = cache miss → normal flow continues
    except Exception as e:
        logger.error(f"Cache get error: {e}")
        return None              # Never raise — cache failure ≠ system failure
```

**Contrast with critical service failure:**
```python
# In chat router — RabbitMQ failure IS fatal for this endpoint
if not rabbitmq_service.is_available():
    raise HTTPException(
        status_code=503,
        detail="Message queue unavailable. Please try again later."
    )
    # Cannot proceed without queue — tell client explicitly
```

## 4.1.5 The Reconnect Pattern (ChatWorker)

```python
# chat_worker.py — _run_consumer()
def _run_consumer(self):
    while self._running:
        try:
            self._connect()                # Establish connection
            self._channel.start_consuming()  # Blocking — runs until exception
        except pika.exceptions.AMQPConnectionError as e:
            if self._running:
                logger.warning(f"RabbitMQ connection lost: {e}. Retrying in 5s...")
                time.sleep(5)              # Wait before retry
                continue                  # Try again
        except Exception as e:
            if self._running:
                logger.error(f"Unexpected worker error: {e}")
                time.sleep(5)
                continue
        finally:
            self._cleanup_connection()    # Always clean up
```

**Key principles:**
1. **Catch specific exceptions first** (AMQPConnectionError) then broad (Exception)
2. **Check `self._running`** — don't retry if stop() was called
3. **Always clean up** in `finally` — avoid connection leaks
4. **Exponential backoff** in production (5s → 10s → 20s) to avoid hammering a recovering broker

## 4.1.6 Error Response Structure

Your system should always return structured errors to clients:

```python
# FastAPI returns this format automatically for HTTPException
{
    "detail": "Message queue unavailable. Please try again later."
}

# For validation errors (Pydantic 422):
{
    "detail": [
        {
            "loc": ["body", "message"],
            "msg": "field required",
            "type": "value_error.missing"
        }
    ]
}

# For WebSocket — text message with error flag:
{
    "type": "error",
    "message": "An error occurred processing your request",
    "job_id": "abc123"
}
```

## 4.1.7 Logging at the Right Level

```python
import logging
logger = logging.getLogger(__name__)

# DEBUG — detailed execution trace (dev only, too verbose for production)
logger.debug(f"Processing job {job_id}, query length: {len(query)}")

# INFO — normal operations worth recording
logger.info(f"Job {job_id} completed successfully in {elapsed:.2f}s")

# WARNING — something unexpected but recoverable
logger.warning(f"Redis cache unavailable, proceeding without cache")

# ERROR — operation failed, needs attention
logger.error(f"Job {job_id} failed: {e}", exc_info=True)  # exc_info adds traceback

# CRITICAL — system is failing, immediate action needed
logger.critical(f"RabbitMQ permanently disconnected, workers stopped")
```

## 4.1.8 Interview Questions and Answers

**Q: What happens when the RAG processing raises an unhandled exception in the worker?**

A: The worker's `_process_job()` method is wrapped in a try/except. Any exception is caught, logged with a traceback, and an error response message is published to the Redis channel for that job/session. The WebSocket handler waiting on that channel receives the error message and sends it to the client. The RabbitMQ message is then NACK'd with `requeue=False`, preventing infinite retry loops.

**Q: Why is the Redis cache configured to fail silently instead of raising exceptions?**

A: The cache is an optimization layer — its failure should not break the core functionality. If the cache is down, the system falls back to the slower but functional path (re-computing the response). Raising an exception would cause FAQ queries to completely fail even though the database is fine. This is the graceful degradation pattern: optional components fail without cascading.

**Q: What would you change to add retry logic for transient job failures?**

A: For retryable failures (e.g., temporary OpenAI rate limiting), instead of `basic_nack(requeue=False)`, you'd implement: 1) Check the delivery count header (x-death count), 2) If below max retries, `basic_nack(requeue=True)` or publish to a retry queue with a delay, 3) If at max retries, move to DLX. You'd also add a Dead Letter Exchange to capture permanently failed messages for inspection.
